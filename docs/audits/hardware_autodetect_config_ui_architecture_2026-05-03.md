# Hardware Autodetect / Config / UI Architecture Note

Created: 2026-05-03 Europe/Berlin
Repository: kosred/forex-ai
Scope: replace env-driven runtime behavior with hardware autodetection, generated config, and UI-managed settings.

## User idea being recorded

The user proposed that most runtime config currently controlled by env vars should instead be generated automatically from hardware detection during program installation, because the bot will be distributed as an installable application.

The user also proposed that the installer should create the initial config and that later changes should be possible from the application UI settings.

This is a good direction and should become part of the architecture.

## Core principle

Env vars should not be the primary way to control strategy/search/model/backtest behavior.

The target should be:

```text
install-time hardware detection
-> generated hardware profile
-> generated default runtime config
-> UI-editable settings
-> typed execution plan
-> artifact provenance
```

This is better than scattered env vars because it makes runtime behavior visible, reproducible, testable, and user-editable.

## Why this matters

The current codebase uses many env vars for behavior that affects results, speed, device choice, fallback behavior, precision, budgets, kernel units, and CPU/GPU execution.

Examples seen during audit include:

- CUDA device selection
- kernel enable/disable flags
- kernel units
- search precision
- training precision
- backtest initial equity
- max month bucket capacity
- GPU-required fail-fast behavior
- neuro-evolution max evaluations
- CPU thread count

These are not just operational toggles. Many of them affect semantics and reproducibility.

A strategy discovered with one env setup may not be equivalent to the same run under another env setup.

## Proposed architecture

### 1. Install-time hardware detection

During installation or first launch, the program should detect available hardware:

- CPU model
- CPU core/thread count
- RAM
- NUMA topology when available
- CUDA devices
- VRAM per GPU
- CUDA capability / compute capability
- GPU count
- GPU memory bandwidth if detectable
- peer-to-peer / NVLink availability when applicable
- PCIe grouping / topology if detectable
- supported precision modes
- available acceleration backends: CUDA, ROCm, WGPU, Vulkan, Metal, CPU
- disk type and free space

The output should be stored as a hardware profile file.

Example:

```text
config/hardware_profile.json
```

### 2. Optional hardware benchmark

Hardware detection should ideally include a small benchmark suite:

- feature build throughput
- signal synthesis throughput
- backtest batch throughput
- model inference throughput
- model training throughput where cheap enough
- host-to-device transfer throughput
- device-to-host transfer throughput

This benchmark should not be mandatory for every launch, but it is useful at install time and after hardware changes.

Example output:

```text
config/hardware_benchmark.json
```

### 3. Generated default runtime config

After hardware detection, the installer should generate a default runtime config:

```text
config/runtime.toml
```

This config should include:

- preferred backend
- available devices
- default GPU assignment policy
- max concurrent GPU jobs
- CPU thread pool size
- search population defaults
- search chunk size defaults
- backtest batch size defaults
- feature build batch size defaults
- model training batch sizes
- precision policy
- fallback policy
- memory budget per device
- disk cache locations
- checkpoint frequency

### 4. UI-managed settings

The application UI should expose safe settings.

The user should be able to change:

- GPU-first / balanced / CPU-only mode
- which GPUs are enabled
- memory budget per GPU
- precision preference
- max CPU threads
- max parallel search jobs
- max parallel training jobs
- checkpoint interval
- fallback policy
- power/temperature profile if supported
- search aggressiveness profile
- artifact retention policy

These UI settings should write to a config file, not to env vars.

Example:

```text
config/user_settings.toml
```

### 5. Layered config model

Use a layered config model:

```text
Built-in safe defaults
< hardware_profile.json
< hardware_benchmark.json
< runtime.toml generated by installer
< user_settings.toml changed by UI
< explicit CLI args for one run
```

The resolved result should be a typed immutable runtime plan for that run.

Example:

```rust
pub struct ResolvedRuntimeConfig {
    pub hardware_profile_id: String,
    pub execution_mode: ExecutionMode,
    pub devices: Vec<DeviceAssignment>,
    pub precision_policy: PrecisionPolicy,
    pub fallback_policy: FallbackPolicy,
    pub search_policy: SearchPolicy,
    pub backtest_policy: BacktestPolicy,
    pub model_policy: ModelPolicy,
    pub checkpoint_policy: CheckpointPolicy,
}
```

## GPU-first execution modes

The UI should not expose low-level env flags. It should expose clear modes:

```rust
pub enum ExecutionMode {
    GpuFirst,
    GpuRequired,
    Balanced,
    CpuOnly,
    DebugCpuReference,
}
```

Meaning:

- `GpuFirst`: use GPU for all supported heavy workloads, CPU as fallback only when explicitly allowed.
- `GpuRequired`: fail closed if required GPU kernels are unavailable.
- `Balanced`: use GPU where useful, avoid oversubscription, allow CPU fallback.
- `CpuOnly`: deterministic CPU reference / low-hardware mode.
- `DebugCpuReference`: CPU parity/debug mode.

## Fallback policy

Fallback must be explicit and visible.

```rust
pub enum FallbackPolicy {
    FailClosed,
    AllowCpuFallbackWithWarning,
    AllowHybridGpuCpu,
    CpuReferenceOnly,
}
```

The app should never silently turn a GPU-required run into a CPU run.

## Precision policy

Precision should be typed:

```rust
pub enum PrecisionPolicy {
    Fp32Canonical,
    Bf16PreferredWithFp32Fallback,
    Fp16PreferredWithFp32Fallback,
    ExperimentalLowBit,
}
```

Approximate precision should not be confused with canonical validation.

## Scheduler-generated device assignment

Kernel files should not read env vars to choose devices.

Instead, the scheduler should pass:

```rust
pub struct DeviceAssignment {
    pub backend: AcceleratorBackend,
    pub device_ids: Vec<usize>,
    pub precision: TrainingPrecision,
    pub memory_budget_gb: Option<f64>,
    pub kernel_units: Option<u32>,
}
```

This should replace scattered env vars such as:

- `FOREX_BOT_SEARCH_EVAL_CUDA_DEVICE`
- `FOREX_BOT_NEAT_CUDA_DEVICE`
- `FOREX_BOT_NEURO_EVO_CUDA_DEVICE`
- `FOREX_BOT_SEARCH_GPU_KERNEL_UNITS`
- `FOREX_BOT_NEAT_KERNEL_UNITS`
- `FOREX_BOT_NEURO_EVO_KERNEL_UNITS`

## What env vars may still be used for

Env vars are still acceptable for:

- logging level
- CI behavior
- test-only overrides
- secrets / credentials
- emergency fail-fast switches
- temporary developer diagnostics

Env vars should not define normal production semantics for:

- model behavior
- strategy behavior
- search fitness
- backtest settings
- device assignment
- precision selection
- fallback behavior
- training budgets
- evaluation budgets

## Artifact provenance requirement

Every artifact should store the resolved runtime plan used to produce it.

At minimum:

- hardware profile ID
- config hash
- user settings hash
- device assignments
- effective backend
- effective precision
- execution mode
- fallback policy
- runtime mode
- canonical vs approximate mode
- dataset fingerprint
- feature schema hash
- code version / git commit

This makes a saved strategy auditable and reproducible.

## Installation / first-launch flow

Recommended flow:

```text
1. Install application
2. Run hardware detection
3. Run optional small benchmark
4. Generate hardware_profile.json
5. Generate runtime.toml
6. Show detected hardware in UI
7. Let user choose execution profile
8. Save user_settings.toml
9. Start first run using typed ResolvedRuntimeConfig
```

## UI settings examples

Suggested UI sections:

### Hardware

- detected CPU
- detected RAM
- detected GPUs
- VRAM per GPU
- backend availability
- benchmark status
- rerun hardware detection button

### Performance

- execution mode
- max concurrent GPU jobs
- max CPU threads
- memory limit per GPU
- checkpoint frequency

### Search

- search intensity: conservative / normal / aggressive / extreme
- max population
- max generations
- candidate validation mode
- approximate presearch enabled/disabled

### Safety / correctness

- require canonical validation
- allow approximate presearch
- CPU/GPU parity test mode
- fail closed if GPU unavailable

### Artifacts

- artifact directory
- retention policy
- include full provenance
- checkpoint resume mode

## Correctness requirement

The UI may allow performance tuning, but it must not allow users to accidentally bypass canonical validation.

Approximate GPU presearch can be enabled, but final strategy acceptance must require canonical validation.

## Bottom line

The user's idea is correct and should be adopted.

The project should move away from scattered env vars and toward install-time hardware autodetection, generated runtime config, UI-managed settings, and typed scheduler execution plans.

This improves:

- reproducibility
- user experience
- debuggability
- hardware portability
- artifact trustworthiness
- CPU/GPU parity testing
- multi-GPU scheduling

For a distributed application, this is the right architecture.
