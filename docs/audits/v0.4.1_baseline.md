Canonical stages (in order):

|---|------------|

### Existing nonzero-signal instrumentation

`crates/forex-search/src/discovery.rs:1907-1925` — there is already
an **aggregate atomic counter** that records how many genes generated
any non-zero signal at all:

```text
let nonzero_signal_count = std::sync::atomic::AtomicUsize::new(0);
let signals_with_idx: Vec<(usize, Gene, Vec<i8>)> = prefiltered
    .into_par_iter()
    .filter_map(|(candidate_idx, gene)| {
        let sig = signals_for_gene_full(features, ohlcv, &gene, &eval_config_for_signals);
        let trade_count = sig.iter().filter(|v| **v != 0).count() as f64;
        if trade_count > 0.0 {
            nonzero_signal_count.fetch_add(1, std::sync::atomic::Ordering::Relaxed);
        }
        if trade_count >= min_trades as f64 {
            Some((candidate_idx, gene, sig))
        } else {
            None
        }
    })
    .collect();
```

The feature-frame summary at `discovery.rs:1786-1827` already emits
**per-frame** statistics (NaN frac, zero frac, min/max finite,
mean_abs_finite) to the `forex_search::funnel` tracing target. So the
tracing infrastructure is in place; what's missing is the **per-gene**
granularity inside the stage-9 hot loop.

### What's missing for the v0.4.2 root-cause diagnosis

The aggregate counter answers "how many genes failed?" but **not**:

- Which specific gene signatures failed.
- Whether `signals_for_gene_full` returned a vector full of zeros, a
  vector full of NaNs that got silently filtered down to zeros, or a
  vector with a few sparse signals below `min_trades`.
- Whether the gene's `long_threshold` / `short_threshold` exceeded the
  max combined-signal magnitude on the feature frame (the
  "threshold > max signal → never fires" failure mode flagged in the
  comment at `discovery.rs:1902-1906`).

### Proposed v0.4.1 patch (no behaviour change)

Add a structured per-gene record emitted alongside the existing
funnel JSON. Suggested shape, one JSONL line per filtered gene:

```jsonc
{
  "gene_idx": 12,
  "strategy_id": "ema9-sma21-...",
  "long_threshold": 0.30,
  "short_threshold": -0.30,
  "signal_len": 18432,
  "nonzero_count": 0,
  "nan_count": 0,
  "min_signal_abs": 0.0,
  "max_signal_abs": 0.18,
  "passed_min_trades": false,
  "min_trades_required": 124
}
```

Output path: `<artifact_dir>/<symbol>_<tf>_gene_signals.jsonl` (same
directory the funnel JSON already lands in). Existing `nonzero_signal_count`
atomic stays — the JSONL is purely additive.

The instrumentation pass should be **gated behind a config flag**
(suggested: `runtime_overrides.dump_gene_signals: bool, default false`)
so production runs don't pay the per-gene serialisation cost.

### Breakdown by category

- **Feature-gated (legit) — 14 sites:** lightgbm (4), catboost (4),
  xgboost (2), trading.rs cfg_attr(not(test)) (2), broker_config.rs
  cfg_attr(not(test)) (1), plus the 2 generated-protobuf module-level
  suppressions on ctrader_proto_messages.rs + ctrader_session.rs.
  These stay until the owning feature/boundary changes.
- **Real debt — 10 sites:** swarm_impl.rs (4), ctrader_bootstrap.rs (2),
  ctrader_execution.rs (1), ctrader_data.rs (1), theme.rs (1),
  ctrader_proto_messages.rs (... actually counted in legit above —
  this is the module-level for generated code).

  `forex-models/src/forecasting/swarm_impl.rs` lines 1366, 1989, 2371,

  marked items or call them.
- Convert `ctrader_execution.rs:591`, `ctrader_data.rs:553`,
  `ctrader_bootstrap.rs:191`, `ctrader_bootstrap.rs:287` to either
  precise `cfg_attr(feature = "ctrader-future-methods", ...)` gates or

- `theme.rs:308` — survey which constants are actually unused vs which

  `#[allow(dead_code)]` block-level justification comment for the
  retained ones.
- **Goal:** drop the 24-site count to **≤ 14** (only the legit
  feature-gated ones remain).

## 3. Version drift — Cargo.toml × 7 stuck at 0.2.0

`grep '^version' Cargo.toml crates/*/Cargo.toml`:

```
crates/forex-cli/Cargo.toml:3:version = "0.2.0"
crates/forex-data/Cargo.toml:3:version = "0.2.0"
crates/forex-news/Cargo.toml:3:version = "0.2.0"
crates/forex-app/Cargo.toml:3:version = "0.2.0"
crates/forex-models/Cargo.toml:3:version = "0.2.0"
crates/forex-core/Cargo.toml:3:version = "0.2.0"
crates/forex-search/Cargo.toml:3:version = "0.2.0"
```

**All 7 crates report `0.2.0` from `CARGO_PKG_VERSION`** even though
`master` is tagged `v0.4.0`. Evidence: `docs/qa/2026-05-12-gui-test-report.md:3`
records `target/release/forex-app.exe ... version 0.2.0` on the v0.4.0
build.

User-visible impact: the GUI status bar (`forex-app/src/main.rs:582-651`,
which uses `env!("CARGO_PKG_VERSION")`) shows `v0.2.0` to end-users
running v0.4.0.

**Patch in this PR:** bump all 7 crate versions to `0.4.1`. Cargo.lock
auto-regenerates on next `cargo check` (deferred to VPS). The Cargo.toml
edit is mechanical and safe to make without a local build.

(Workspace-level version inheritance via `workspace.package.version`
is a deferred refactor — out of scope here.)
