### The headline bug — EURJPY/XAUUSD empty portfolios

The forex-search agent traced the suspected root cause to **three combined
sites** in `discovery.rs`:

1. **F-SEARCH-001** (CRITICAL) — `genetic/search_engine.rs:145-151` —
   genes with out-of-bounds feature indices silently `continue` without
   rejecting the gene, so they pass through fitness evaluation as
   near-zero-signal strategies.
2. **F-SEARCH-002** (CRITICAL) — `discovery.rs:1484-1509` — the feature
   prefilter ranks features using forward returns computed over the **full
   dataset** including the OOS tail; archived genes are biased toward
   features that work post-OOS.
3. **F-SEARCH-003** (CRITICAL) — `discovery.rs:1024-1095` — empty
   portfolios return `Ok(pending_gates)` instead of `Err("portfolio is
   empty")`, so operators see "validation gates not met" (ambiguous)
   instead of "portfolio generation failed" (actionable).

These three together explain the observed behaviour: bad gene candidates
survive (F-SEARCH-001), the prefilter compounds the bias (F-SEARCH-002),
and the empty result is reported as a gate failure (F-SEARCH-003)
instead of the actual root cause.

### Severity definitions

- **CRITICAL** — silent correctness failure that can corrupt P&L,
  bypass risk limits, leak future data into training, lose user money,
  or panic the GUI thread.
- **HIGH** — silent failure on the error path, missing tracing on
  silent fallbacks, race-conditions in non-mission-critical loops,
  panics in hot paths.
- **MEDIUM** — missing input validation, silent UX failures,
  inconsistent error handling, performance hazards.
- **LOW** — dead code, cosmetic, hard-coded magic numbers, doc gaps.

## §1 — forex-app (42 findings)

**Architecture summary:** Rust/egui GUI for cTrader trading. Core
weakness: `TradingSession` is a 3,372-line god-object handling auth,
execution, market data, caching, and risk gating — tightly coupled to
`AppState` via direct mutation.

### CRITICAL findings

- **F-APP-001 | CRITICAL | Concurrency** — `app_services/ctrader_execution.rs:525-527` — mutex poisoning converted to generic `anyhow!` error; UI cannot offer "restart required" recovery. Fix: dedicated `ExecutionError::MutexPoisoned` variant.
- **F-APP-002 | CRITICAL | FP correctness** — `app_services/trading.rs:2290-2310` — drawdown checks use `>=` on `f64` ratios without epsilon. A `10000 * 0.04 = 400.0000000001` rounding error blocks legitimate trades. Fix: `(ratio - limit).abs() < EPSILON`.
- **F-APP-003 | CRITICAL | UI panic** — `ui/theme.rs` — `partial_cmp().unwrap()` on color `f64` slice sorting panics on NaN, crashing the egui render thread. Fix: `unwrap_or(Ordering::Equal)`.
- **F-APP-004 | CRITICAL | Network safety** — `app_services/ctrader_execution.rs:449-459` — socket read-timeout application logs a warn and proceeds with no timeout if `.set_read_timeout()` fails. Trading thread wedges indefinitely on broker stall. Fix: return error; let caller decide retry vs fail.
- **F-APP-005 | CRITICAL | OAuth safety** — `app_services/trading.rs:2142-2165` — refreshed token bundle is persisted before validating it. If persist fails, in-memory holds new token while disk has stale token; next restart loses it. Fix: validate non-empty + later expiry before persisting.
- **F-APP-006 | CRITICAL | Order idempotency** — `app_services/ctrader_execution.rs:533-589` — retry loop preserves same clientOrderId across auth resets; broker's idempotency is per-session, so duplicate orders can submit under different auth keys. Fix: increment client message ID on auth-failure retry.
- **F-APP-007 | CRITICAL | Time math** — `app_services/trading.rs:1644-1665` — day ID = `broker_now_unix_ms / 86_400_000` ignores broker timezone. NY 11 PM Tuesday (UTC 4 AM Wednesday) attributed to wrong calendar day. Daily-DD reset at wrong wall clock. Fix: accept `broker_timezone_offset` parameter.
- **F-APP-008 | CRITICAL | Money math** — `app_services/ctrader_execution.rs:769-776` — missing `money_digits` defaults to 2 with only a tracing::error!. If stderr is misconfigured, operator never sees the warning and P&L reports off by 100x. Fix: hard-error if missing, force broker-adapter update.
- **F-APP-009 | HIGH | Env-var trap** — `app_services/ctrader_execution.rs:980-1000` — partial fills silently re-enabled via `FOREX_BOT_CTRADER_ALLOW_PARTIAL_FILL=1` with no UI indicator. Fix: surface as Risk-panel toggle + WARN log on every partial accept.

### HIGH findings (summarised)

- F-APP-010 — `trading.rs:58` — `CTRADER_TOKEN_REFRESH_WINDOW_SECS = 300` hardcoded; no env override.
- F-APP-011 — `trading.rs:1852-1879` — live spot cache key omits bid/ask; stale quotes returned within 1-second TTL.
- F-APP-012 — `trading.rs:1595-1611` — unrealized PnL == 0 with open positions silently uses balance-only equity for daily-DD check.
- F-APP-013 — `trading.rs:1481-1504` — smart-SL falls through to `None` if <14 candles loaded; user expects ATR-based SL.
- F-APP-014 — `trading.rs:1439-1464` — NaN ATR → NaN multiplier → cast to i64 is UB; SL=0 order may submit.
- F-APP-015 — `trading.rs:1644-1665` — `day_start_equity` accepts negative values; daily-DD bypassed once account goes negative.
- F-APP-016 — `ctrader_execution.rs:240-276` — close-position fingerprint omits sequence number; double-click in 1s returns first outcome.
- F-APP-017 — `trading.rs:841-906` — market-chart snapshot rebuilds on every 16ms render even when cache fresh; `clone()` per frame.
- F-APP-018 — `trading.rs:1690-1717` — `refresh_runtime()` exits early on disconnect, leaving stale `account_balance`.
- F-APP-019 — `trading.rs:1747-1754` — trade-journal capped at 16 with silent FIFO eviction; no persistence of evicted lines.
- F-APP-020 — `trading.rs:3134-3140` — `(volume * 100).round() as i64` loses precision; 0.115 lots → 12 (not 11.5).
- F-APP-021 — `trading.rs:909-951` — `execution_surface_snapshot()` rebuilds positions list every frame (O(n) inside O(60Hz)).
- F-APP-022 — `app_services/broker_persistence.rs` — silent broker-settings load failure at startup; no recovery UI offered.

### MEDIUM findings (summarised)

F-APP-023 through F-APP-038 — covers: smart-SL default `true` (UX risk), 30s socket timeout default too high, blocking heartbeat send,
order-type state machine without validation, account sync deduplication
gap, risk rules not retroactive on config change, ATR 13/14-bar comment
inconsistency, fallback pip-value heuristic uses empty symbol, OAuth
errors not differentiated for UI, deprecated `connect()` method retained,
tight coupling (TradingSession ↔ AppState mutation), no auto-reconnect
on broker drop, no structured tracing on risk-gate rejections, duplicate
account-sync logic in two directions, duplicate `selected_ctrader_*_account_id`
methods (chart vs execution).

### LOW findings (summarised)

F-APP-039 through F-APP-042 — `scaled_money()` lacks bounds doc, no
day-boundary integration tests, no FP-edge-case tests for risk gates,
unnecessary `Arc`-cloning vs `clone()` on chart snapshots.

## §2 — forex-cli (32 findings)

**Architecture summary:** Dual-mode binary — batch CLI (~15 subcommands)
and interactive TUI (ratatui + crossterm, 9 pages with subprocess
spawning). Core weakness: subprocess argv construction loosely coupled
to subcommand contract; multiple silent-fallback paths in config / data
root resolution.

### CRITICAL findings

- **F-CLI-001 | CRITICAL | Concurrency** — `main.rs:710-712` — `unsafe { std::env::set_var("FOREX_BOT_DATA_ROOT", &root) }` mutates global env with comment "single-threaded; no other thread reads env" — but `cmd_train` spawns subprocesses that inherit env race-conditionally. Fix: pass `--root` to `cmd_train` like `cmd_discover` already does.
- **F-CLI-002 | CRITICAL | Time** — `tui/app.rs:452-458` — `unix_seconds.unwrap_or(0)` falls back to `00:00:00 UTC` on `SystemTime` error and stays there until next frame. Fix: use `Instant`-based monotonic time + show error indicator.
- **F-CLI-003 | CRITICAL | Path traversal** — `tui/form.rs:182-197` — `discover_symbols_in_root` accepts `symbol=../../../etc` directory names without validation; attacker controlling data root can expose arbitrary paths in TUI selector. Fix: validate stripped name is a single Path component.
- **F-CLI-004 | CRITICAL | Argv hygiene** — `tui/pages/discover.rs:320-332` — user-typed symbols/timeframes passed directly to argv with no whitespace/comma validation. Safe via no-shell `Command`, but malformed strings fail cryptically downstream. Fix: validate `[A-Z0-9]+` symbols + known-timeframe list before spawn.
- **F-CLI-005 | CRITICAL | Config fallback** — `main.rs:125-126` — `default_symbol(settings)` returns empty string if `settings.system.symbol == ""`, leading downstream to load `symbol=""` data with silent failure. Fix: validate non-empty in `Settings::from_yaml()`.

### HIGH findings (summarised)

F-CLI-006 to F-CLI-013 — subprocess spawn error truncation, form-field
panic on out-of-bounds focus, ring-buffer silent overflow, race-condition
on shared `forex-ai.log` read, silent JSON deserialisation failure on
`AutoLoopCheckpoint`, NaN propagation in sparkline (already clamped, but
fragile), hardcoded cache paths with no override, dataset-summary
filesystem walk every render.

### MEDIUM findings (summarised)

F-CLI-014 to F-CLI-027 — hardcoded fallback symbol/data-dir defaults,
unvalidated path fields, `disable_raw_mode().ok()` silently fails on
TUI exit, `event::poll().unwrap_or(false)` silently breaks event loop,
`stop_editing(commit)` parameter unused (revert-on-Esc not implemented),
deadlock-risk on `JobManager` channel if receiver dropped, loose coupling
to `batch-discover` subcommand contract, Unicode width not accounted for
in tab hit-rects, `active_job_count() -> 0` placeholder, dead `_label_value()`
function, `FormState.message` write-only from page code.

### LOW findings (summarised)

F-CLI-028 to F-CLI-032 — implicit Settings::load validation,
redundant `.clone()` on sparkline, missing docstring on
`strip_ansi_for_display()`, timezone-naive log-page reading entire
file every frame, no CLI integration tests.

## §4 — forex-data (30 findings)

**Architecture summary:** OHLCV ingestion, feature engineering (SMC,
quant, session, regime), normalisation, Parquet/Vortex IO. Core weakness:
multiple silent-default paths (volume=0 fallback, CSV missing-column→0.0,
NaN→0.0 in normalisation) + lookahead bias in SMC features.

### CRITICAL findings

- **F-DATA-001 | CRITICAL | Data integrity** — `core/lib.rs:227` — `row.volume.unwrap_or_default()` silently produces 0.0 for missing volume; downstream VPIN/Kyle's Lambda/Amihud train on synthetic zeroes. Fix: track missingness separately; return error if strategy requires volume.
- **F-DATA-002 | CRITICAL | Overflow** — `core/resample.rs:25` — `mins * 60 * 1_000_000_000` overflows i64 for `mins > ~9M`. Fix: `checked_mul` chain.
- **F-DATA-003 | CRITICAL | Lookahead bias** — `core/lib.rs:383-430` + `core/smc.rs:343-375` — SMC fractal swing detection uses `center ± SWING_FRACTAL` (5 bars forward) to identify the current bar's swing. GA overfits on unachievable real-time signals. Fix: enforce causal window — only past bars allowed.
- **F-DATA-004 | CRITICAL | NaN propagation** — `core/smc.rs:163-165` — ATR computed without NaN guard, propagates into gap_size / running_atr.max(1e-10) which doesn't trap NaN. Fix: validate input OHLCV NaN-free before `compute_smc_feature_columns`.
- **F-DATA-005 | CRITICAL | Timezone** — `core/session_features.rs:147-199` — `(7..16).contains(&hour)` hard-coded for London-UTC; USDJPY on JST broker computes London-session features on JST 16:00. Fix: accept `session_tz_offset_hours` parameter.
- **F-DATA-006 | CRITICAL | CSV parse** — `core/universal_importer.rs:456-468` — `parse_f64(rec.get(*col).unwrap_or("0"))` silently defaults missing OHLC columns to 0.0; validate_ohlcv_row runs post-import, doesn't catch the synthetic zero. Fix: return error on missing mandatory columns.
- **F-DATA-007 | HIGH** — `core/timestamps.rs:71-77` — some paths in unit conversion use `checked_mul`, others don't.

### HIGH findings (summarised)

F-DATA-008 to F-DATA-013 — resample bucket assignment off-by-one
(`div_euclid` vs `>`/`>=`), volume=0 not validated in volume-weighted
indicators, unsafe lossy `f64 → f32` cast in FeatureFrame (EURJPY can
exceed f32 range), MTF alignment doesn't validate monotonic timestamps,
**FVG detection always overwrites bearish with bullish in same bar**
(`smc.rs:477-493`) — bearish FVG features are dead code, Garman-Klass
volatility produces NaN on prices < 1e-10.

### MEDIUM findings (summarised)

F-DATA-014 to F-DATA-022 — Fibonacci proximity uses sharp `(1 - 15*dist).max(0)`
cutoff (discontinuity), timestamp unit inferred from FIRST non-zero
sample (mixed-unit arrays misidentified), session reset triggered twice
on duplicate midnight bars, volume-imbalance loop has `if i >= 1` inside
`for i in 0..n` (index 0 always skipped), consecutive-bar counter resets
on dojos, Vortex round-trip verifies only row count not content, month-key
calculation lacks `checked_div_euclid`, NaN→0.0 normalisation silently
loses signal, session-open-gap broadcast only at exact boundary minute.

### LOW findings (summarised)

F-DATA-023 to F-DATA-029 — swing_highs computed but discarded (dead
code), unused `_consol_range_avg`, inefficient per-column allocation
in HPC TA, quant features omitted when volume None (inconsistent feature
frame shape), magic numbers in SMC thresholds, Parquet migration doesn't
verify column types, hardcoded SMC_FEATURE_NAMES const drifts from
implementation.

## §5 — forex-models (30 findings)

**Architecture summary:** Largest crate (47k LOC). Six model families
(tree / deep / RL / forecasting / streaming / evolution) coordinated by
`training_orchestrator.rs` (4,077 LOC god-file). Core weakness:
**multiple data-leakage vectors** across feature selection, label
generation, train/val split, normalisation — difficult to enforce
consistently across model families.

### CRITICAL findings

- **F-MODELS-001 | CRITICAL | ML correctness** — `training_orchestrator.rs:868-961` — L1 feature selection runs on full training window (first 80%) but downstream training applies budget-clip and resplit without re-embargo. Selected features may be biased toward held-out window. Fix: re-apply embargo + resplit AFTER feature selection.
- **F-MODELS-002 | CRITICAL | Label leakage** — `training_orchestrator.rs:1798-1914` — triple-barrier labels look forward to `(i + hold_bars).min(n-1)`. Rows near dataset end implicitly know "future doesn't extend further" (test-set bias). Fix: `horizon_end < (n - embargo)` hard floor.
- **F-MODELS-003 | CRITICAL | Normalisation leakage** — `training_orchestrator.rs:268-285` — features normalised on full clean payload before being passed to models; per-split normalisation must be applied INTERNALLY by each model, but contract isn't enforced. Fix: assert model receives raw features; centralise scaling at trait level.
- **F-MODELS-004 | CRITICAL | RL look-ahead** — `rl/dqn_impl.rs:676-702` — `build_reward_triplet()` computes per-action rewards by examining next `horizon` bars from state `idx`. State[idx] thus knows future rewards. Production = catastrophic. Fix: store single immediate reward + bootstrap from `Q(next_state)`.
- **F-MODELS-005 | CRITICAL | Artifact versioning** — `forecasting/swarm_impl.rs:110-125` — multiple `#[serde(default)]` fields silently fill with 0.0 on missing input. Old artifact loaded by new code → silent model degradation. Fix: explicit `artifact_version: u32` + fail-hard on incompatible version.
- **F-MODELS-006 | CRITICAL | GPU unsafe** — `statistical/linear_gpu.rs:381-426` — `ArrayArg::from_raw_parts` borrowed across multiple async kernel launches with no `client.sync()` between them; potential use-after-free. Fix: explicit sync points; document handle lifetime.
- **F-MODELS-007 | CRITICAL | Histogram edge** — `base.rs:974-980` — bin assignment uses `<` / `>=` without epsilon; ties on breakpoint values mis-assigned. Fix: `total_cmp()` + consistent tie-break + assert strict monotonic breakpoints.
- **F-MODELS-008 | CRITICAL | Feature scaling drift** — `tree_models/common.rs` + Burn deep models — tree models expect raw features; Burn may apply internal scaling. Cross-model contract not enforced. Fix: `requires_raw_features: bool` flag on `ExpertModel` trait.

### HIGH findings (summarised)

F-MODELS-009 to F-MODELS-016 — Swarm ensemble silently falls back to
baseline if all forecasts NaN (no error signal), `stratified_downsample`
seed reuse breaks CV independence, fallback Q-function weights
serialised without scaler metadata, embargo floor read from env var
not artifact, off-by-one in episode window slicing, missing bounds check
in seasonal forecast (can panic on small data), silent artifact repair
masks corruption/version mismatch, feature-selection embargo not
re-validated post-prefilter.

### MEDIUM findings (summarised)

F-MODELS-017 to F-MODELS-024 — Halton sequence precision loss at high
trial indices, feature-dim mismatch only caught at inference, HPO best-trial
selection non-deterministic (no epsilon in score comparison), validation
windows lack explicit embargo between train and val, Transformer "replicas"
have varying dropout (not true ensemble members), fallback basis expansion
can produce NaN/Inf on large features, discretisation can produce
out-of-bounds bin index due to FP rounding, `TrainingPayload` doesn't
assert temporal order of rows.

### LOW findings (summarised)

F-MODELS-025 to F-MODELS-030 — confidence threshold clamp allows 1.0
(filters all predictions), online learners can div-by-zero on empty
batch, unnecessary `.clone()` in feature ranking, hardcoded swarm window
sizes, no GA fitness memoisation, adaptive LR lacks null-batch check.

## §6 — forex-news (6 findings, manual audit)

**Architecture summary:** Tiny crate (132 LOC, 3 files) — OpenAI sentiment
scoring + Perplexity news search. Both stateless `reqwest::Client`
wrappers with `Bearer` auth.

### Findings

- **F-NEWS-001 | HIGH | Silent misconfiguration** — `openai.rs:14` + `perplexity.rs:14` — `env::var("OPENAI_API_KEY").unwrap_or_default()` silently makes the key empty, then `analyze_sentiment()` returns `Ok(0.0)` on every call. The trader sees neutral sentiment that doesn't exist. Fix: `Result::Err` from `new()` if key empty, OR loud `tracing::warn!` at boot with exact env var name.
- **F-NEWS-002 | MEDIUM | Silent fallback** — `openai.rs:55` — `content.trim().parse::<f64>().unwrap_or(0.0)` swallows any non-float response from GPT (e.g. "neutral", "+0.3 (medium confidence)") and reports 0.0. Fix: `tracing::warn!` with raw content + retry with stricter prompt.
- **F-NEWS-003 | MEDIUM | No HTTP timeout** — `openai.rs:16` + `perplexity.rs:16` — `Client::new()` has no `.timeout()`. A hung OpenAI request blocks the async task indefinitely. Fix: `ClientBuilder::new().timeout(Duration::from_secs(30))`.
- **F-NEWS-004 | MEDIUM | Credential leakage** — both files — `reqwest`'s default tracing emits the full `Authorization: Bearer ...` header in DEBUG-level logs. Fix: filter `authorization` headers from tracing output OR `Client::builder().sensitive_headers(["authorization"])`.
- **F-NEWS-005 | LOW | Model names hard-coded** — `openai.rs:33` (`gpt-4o-mini`) + `perplexity.rs:33` (`sonar-pro`). When retired they fail silently. Fix: env override (`FOREX_BOT_OPENAI_MODEL`).
- **F-NEWS-006 | LOW | Magic-number tier** — `lib.rs:15` — `tier: i32` with `// 1=High, 2=Med, 3=Low` comment. Fix: `pub enum NewsTier { High = 1, Medium = 2, Low = 3 }`.

## §7 — forex-search (23 findings)

**Architecture summary:** Multi-stage funnel GA. Stages: feature
prefilter → archive → fitness ranking → quality screen → prop-firm gate
→ portfolio assembly → walkforward CV + CPCV → export. Core weakness:
**feature prefilter leaks OOS data into ranking** (F-SEARCH-002) — this
is the single most likely root cause of the EURJPY/XAUUSD empty-portfolio
bug.

### CRITICAL findings

- **F-SEARCH-001 | CRITICAL | GA silent acceptance** — `genetic/search_engine.rs:145-151` — gene with out-of-bounds index `continue`s in weight-accumulation loop, leaving `combined[i] = 0.0`. Gene passes evaluation as near-zero-signal strategy instead of being rejected. Fix: reject genes with OOB indices at construction.
- **F-SEARCH-002 | CRITICAL | Prefilter data snoop** — `discovery.rs:1484-1509` — forward-returns computed over full dataset including OOS tail; comment says "restrict to in-sample" but the actual return computation peeks at `close[i+1]` through entire dataset. Archive survivors 2-30% better than true capability. Fix: compute forward returns only for `i ∈ [0, train_end)`.
- **F-SEARCH-003 | CRITICAL | Silent empty portfolio** — `discovery.rs:1024-1095` — `build_discovery_validation_artifacts()` returns `Ok((pending_gates, empty, empty))` when portfolio empty. Caller sees "gates not met" instead of "portfolio failed". Fix: return `Err("portfolio is empty")`.
- **F-SEARCH-004 | CRITICAL | Prop-firm gate bias** — `discovery.rs:2125-2236` — `pass_rate` ranked but no hard threshold reject; portfolio takes top-N regardless of actual gate score. Fix: `if rate < pf.pass_rate { continue }` before adding to portfolio.
- **F-SEARCH-005 | HIGH | Atomic ordering** — `discovery.rs:1907-1925` — `nonzero_signal_count.fetch_add(_, Relaxed)` inside rayon par-iter; OK for diagnostic but inconsistent with adjacent critical filter logic. Fix: collect-then-sum or `SeqCst`.

### HIGH findings (summarised)

F-SEARCH-006 to F-SEARCH-010 — `FOREX_BOT_DISABLE_SMC_GATE=1` bypass not
recorded in run profile (live bridge can't tell which artifact used
gating), CPCV fold-index offset calculation fragile (no assertion),
walkforward fold silently skipped when `test_start >= end` (no log),
portfolio export gate requires `walkforward_passed && cpcv_passed` but
those flags are never `true` when prop-firm gate alone ran (always
blocks export), profit-factor saturated cap at 100.0 collides with real
values.

### MEDIUM findings (summarised)

F-SEARCH-011 to F-SEARCH-017 — regime robustness silently `return true`
if regime columns missing post-prefilter, `Stage1Window::MostRecent`
warns in enum comment but not at runtime (risk of OOS training), CPCV
embargo by index not time (irregular timeframes break), month/day index
slicing alignment fragile, MC perturbation skips `normalize()` after
mutation (fine today, fragile to future changes), correlation gate
computes over full-length vectors regardless of SMC dropout, consistency
score gives single-month gene the same `1.0` score as 50-month gene.

### LOW findings (summarised)

F-SEARCH-018 to F-SEARCH-020 — unused metric slot at index 7 in
`evaluate_strategy_core` output array, atomic ordering inconsistency
(dup with F-SEARCH-005), walkforward split result fields lack NaN
sanitisation post-construction (currently mitigated by eval-side sanitise).

## §9 — Architecture-wide recommendations

Beyond per-finding fixes, four cross-cutting themes need structural
work:

1. **Validation discipline.** Every layer (config, contracts, OHLCV
   loaders, model artifacts) currently has at least one "warn and
   continue" path where the right behaviour is hard-error. The v0.4.3
   `Settings::validate()` patch should enumerate every cross-field
   invariant and fail load on violation.

2. **Tracing on silent fallbacks.** ~30% of all findings are some
   variant of "code falls back to a default value with no
   `tracing::warn!` or `eprintln!`". A single audit pass adding
   structured warnings to every `unwrap_or_default()`,
   `.unwrap_or(false)`, and `.ok()` would surface most operator
   misconfiguration silently in production today.

3. **Time-series correctness.** The forex-models + forex-data + forex-search
   crates each independently re-implement embargo, train/val split, and
   feature-causality enforcement. A new
   `forex-core::time_series::{Split, Embargo, CausalityGate}` module
   would centralise this — the v0.4.5 dedup patch should land this.

4. **God-file decomposition.** The 5 known god-files
   (`training_orchestrator.rs`, `trading.rs`, `swarm_impl.rs`,
   `discovery.rs`, `dqn_impl.rs`) total ~16,400 LOC — 14% of the entire
   codebase. The v0.4.5 patch's per-file caps (mod.rs, helpers.rs,
   params.rs, runtime.rs) are the right shape but the splits need
   compile-checked re-export contracts to avoid breakage.
