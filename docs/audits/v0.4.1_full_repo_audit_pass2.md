## What Pass 2 added vs Pass 1

Pass 1 had ~183 findings. Pass 2 produced ~277 new + overlapping findings.
The new ones land in five clusters:

### Cluster A — Pass-1 misses now caught (HIGH/CRITICAL)

These are real bugs Pass 1 did not surface because it didn't read the
relevant code:

- **F-APP2-008 CRITICAL** — `ctrader_live_auth.rs:657-666` —
  `build_token_exchange_url()` embeds `client_secret` **in the URL
  query string**. If reqwest debug-logs the URL on any error path, the
  secret leaks into logs in plaintext. Fix: pass secret in request body,
  not URL.
- **F-APP2-014 HIGH** — `ctrader_data.rs:383-393` — tick timestamp delta
  reconstruction is **backwards**: `previous - tick.timestamp` instead
  of `previous + tick.timestamp`. Result: reconstructed historical tick
  timestamps run **backward in time**. Caught only if a downstream
  consumer notices the timestamps don't monotonically increase. This is
  a straight-up bug.
- **F-APP2-101 HIGH** — OAuth callback receives `code` parameter without
  validating the `state` parameter (CSRF protection). Attacker can
  inject an authorization code from a different OAuth session.
- **F-APP2-104 HIGH** — Retry loop reuses `client_order_id` across auth
  resets without first querying the broker for an existing order with
  that ID. Network timeout after broker accept = duplicate order on
  retry.
- **F-CLI2-001 HIGH** — `unsafe { std::env::set_var(...) }` in both
  `main.rs:710` and `tui/pages/train.rs:317`. Comment claims
  "single-threaded" but TUI has rayon + spawn reader threads — actual
  race. (Pass 1 found this; Pass 2 confirmed both call sites.)
- **F-CLI2-003 HIGH** — Form fields (symbols, timeframes, paths) passed
  directly as argv to `Command::new` with no validation. Safe-via-no-shell
  but a malformed input fails cryptically deep in the child process.
- **F-CORE2-027 HIGH** — News-filter API key (`api_key: Option<String>`)
  stored in plaintext memory. Fix: load only from env var, wipe with
  `zeroize` after use.
- **F-MODELS5-002 HIGH** — `onnx.rs:125-138` — `predict_proba()` does
  **not validate** the input feature count vs. the count baked into the
  exported model. Shape mismatch → either silent ORT failure or panic.

### Cluster B — Numerical / FP correctness (MEDIUM)

- **F-MODELS3-006** — Bayesian posterior variance diagonal inversion
  with only a `1e-6` floor → overconfident predictions on rare features.
- **F-MODELS3-012** — Hoeffding fallback quadratic features square
  values without clamping → potential `f32::INFINITY` on large features.
- **F-MODELS3-013** — Online weight update doesn't check `tau.is_finite()`
  before applying → NaN propagation into weights.
- **F-MODELS4-001** — `build_reward_triplet` lookahead truncated near
  episode end → reward asymmetry biases policy toward early-episode
  states.
- **F-MODELS4-011** — NEAT GPU activation code out of [0,9] silently
  returns `0.0` instead of trapping. Corrupted buffer → silent fitness
  degradation.
- **F-MODELS4-012** — CRFMNES GPU has no positive-definite check after
  covariance update → numerical drift can leave Σ singular.
- **F-MODELS5-001** — ONNX output name resolution uses `"prob"`
  substring search with `outputs.last()` fallback. Non-deterministic
  across ORT versions.
- **F-MODELS5-003** — HPO embargo with `saturating_sub` can silently
  truncate `train_end` to 0 if `embargo_rows + val_rows > frame.height()`;
  validation passes if `min_train_rows == 0`.
- **F-CORE2-018** — `storage/json.rs:37-39` fsync of parent dir uses a
  read-only `File` handle. POSIX may not allow fsync on
  RDONLY-opened directory; durability after rename not guaranteed.
- **F-CORE2-031** — Kelly criterion `f_star = (wr*aw - (1-wr)*al) / aw`
  has no epsilon guard on `aw=0`; produces `inf` quietly, then gets
  clamped to `quarter_kelly`. Strategy with no winning trades gets a
  non-zero Kelly fraction.
- **F-CORE2-011** — PSI Laplace smoothing `1e-6` is not bin-count-aware:
  for 1000 bins, `bins as f64 * 1e-6` = 1e-3, which swamps the per-bin
  smoothing for sparse bins.

### Cluster C — Silent fallbacks (MEDIUM/LOW, ~40 records)

Pass 2 found ~40 more silent-fallback findings across `forex-core`,
`forex-cli`, `forex-data`. The pattern repeats: `unwrap_or_default()`,
`unwrap_or(false)`, `.ok()`, `let _ = …`, `silently fall through to
default value`. Examples:

- `F-CORE2-002` — `1.0 / (1.0 + avg_excess_corr)` produces ∞ at `avg_excess_corr = -1.0`.
- `F-CORE2-003` — Portfolio scoring `if all_scores <= 0 { fill 1.0 }`
  without log.
- `F-CORE2-014` — Accelerator backend selection silently downgrades to
  CPU when preferred backend not available.
- `F-CORE2-019` — Pearson correlation silently returns 0.0 on
  zero-variance input.
- `F-DATA2-002` — Active FVGs in SMC kept indefinitely (no max-age) →
  potential lookahead.
- `F-DATA2-019` — Vortex write failure doesn't quarantine source CSV.
- `F-CLI2-005` — Auto-loop checkpoint write isn't atomic (no
  `temp + rename`), TUI reading it can see truncated JSON.
- `F-CLI2-006` — TUI raw mode enabled with no `isatty()` check; piping
  stdout corrupts the output file with ANSI codes.
- `F-CLI2-007` — No SIGTERM/SIGINT handler; killed parent leaves child
  jobs orphaned.

(Full list is in the per-agent reports linked in §6.)

### Cluster D — Conflicts with Pass 1 (REQUIRES MANUAL VERIFICATION)

Pass 2 contradicted Pass 1 in three places. Both passes wrote with
confidence; the conflict means one (or both) is wrong about what the
code actually does. Resolve when VPS is available:

#### Conflict #1 — EURJPY/XAUUSD prefilter leak

- **Pass 1 (F-SEARCH-002, CRITICAL)** claimed: *"comment says 'restrict
  to in-sample' but the actual implementation only limits the
  correlation denominator — not the underlying forward-return
  computation which peeks at `close[i+1]` through the entire dataset."*
- **Pass 2 (F-SEARCH2-005, MARKED FIXED)** claimed: *"The fix is in
  place. Lines 1492-1508 restrict forward-return computation to
  in-sample prefix only; lines 1514-1520 lock regime indicators."*

Both reference the same lines. One of them is reading the code wrong.
The empty-portfolio symptom is concrete and reproducible (EURJPY M30/H1/H4/D1
yield 0 strategies in v0.4.0), so the fact that the bug shows up is
ground truth: **the bug is real**. Either Pass 1's reading of the leak
mechanism is right and Pass 2 misread the fix, or Pass 2's reading is
right and there is a different leak elsewhere. **Action item:** trace
the exact data flow of `forward_returns` from `discovery.rs:1492` end
to end on the VPS, with the existing instrumentation from PR #10's
proposed `gene_signals.jsonl`.

#### Conflict #2 — ML correctness in forex-models

- **Pass 1 (F-MODELS-001, F-MODELS-002, F-MODELS-004, all CRITICAL)** claimed:
  - F-MODELS-001: L1 feature selection runs on full training window
    without post-selection embargo + resplit.
  - F-MODELS-002: Triple-barrier labels look forward to
    `(i + hold_bars).min(n-1)` with no `< (n - embargo)` floor.
  - F-MODELS-004: RL `build_reward_triplet()` examines next `horizon`
    bars from state `idx`, encoding future knowledge into state.
- **Pass 2 (F-MODELS2-005, F-MODELS2-006, F-MODELS2-007, all PASS)** examined
  the same files and called the embargo logic exemplary, citing
  `embargo_rows_for_timeframe()` 20-bar floor.

Both can be partly right: the codebase **has** rigorous embargo logic in
the central paths (`time_series_train_val_split`,
`embargo_rows_for_timeframe`) but Pass 1's specific call-site
observations (e.g. feature selection running on `train_frame` before
budget-clip + resplit) may still be valid. **Action item:** when VPS
is available, run a controlled experiment where the same dataset is
trained with and without the embargo-protected feature-selection path;
compare OOS forward-test metrics.

#### Conflict #3 — Trading.rs god-file decomposition risk

- **Pass 1 (F-APP-034, HIGH)** flagged TradingSession as a 3,372-line
  god-object that needs splitting into AuthManager/OrderExecutor/RiskGate.
- **Pass 2** did not re-surface it because the pt2 agent only got to
  ~2,400 of those 3,372 lines before token-limit cutoff.

This is the only file where the Pass 2 coverage is genuinely partial
(71%). A third pass on just that file is justified if we want airtight
coverage. **Action item:** spawn a `trading.rs`-only agent focused on
the 2400-3372 range OR accept that Pass 1's findings on the file
already cover what an agent could surface.
