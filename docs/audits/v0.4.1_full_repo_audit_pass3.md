## Pass 3 findings — 89 new issues

### Cluster A — New CRITICAL/HIGH from previously-uncovered code

These are issues that Pass 1 + 2 missed because the relevant code was
not read end-to-end. They are confirmed by reading the actual lines:

#### F-APP3-001 CRITICAL — `client_order_id` race via `Ordering::Relaxed`

**File:** `crates/forex-app/src/app_services/trading.rs:2463-2467`
```rust
fn next_client_order_seq() -> u64 {
    use std::sync::atomic::{AtomicU64, Ordering};
    static COUNTER: AtomicU64 = AtomicU64::new(0);
    COUNTER.fetch_add(1, Ordering::Relaxed)
}
```

`Ordering::Relaxed` provides no inter-thread ordering guarantee. Combined
with the second-resolution wall-clock timestamp used elsewhere in the ID
construction (comments at lines 2459-2461), two orders placed milliseconds
apart from different async tasks can theoretically observe identical
`(timestamp_seconds, counter)` tuples → identical `client_order_id` →
broker dedup silently drops the second order, but the local state
considers it placed. Compounds with F-APP2-104 (retry without
broker-side existence check).

**Fix:** `Ordering::SeqCst` (cheap on x86; the only path that increments
this is order-placement, not a hot loop).

---

#### F-APP3-004 CRITICAL — `pip_position` exponent unchecked → Infinity loss in risk gate

**File:** `crates/forex-app/src/app_services/trading.rs:3324`
```rust
let pip_multiplier = 10.0_f64.powi(pip_position);
let pip_distance = (entry_estimate - sl).abs() * pip_multiplier;
```

`pip_position: i32` comes from broker symbol metadata with no upstream
range check. If the broker (or a corrupted symbol entry) returns
`pip_position = 308`, `powi(308) = ∞`. Then
`estimated_loss = ∞`, and the gate at line 3343
(`if estimated_loss > max_loss`) always blocks — risk gating becomes
"all orders rejected". Conversely `pip_position = -308` → underflow
to 0 → `estimated_loss = 0` → all orders pass risk gate regardless
of actual exposure. Both are silent bugs.

**Fix:** Clamp `pip_position` to a sensible range (e.g., `-10..=10`)
or validate symbol metadata at ingest time.

---

#### F-APP3-002 HIGH + F-APP3-003 HIGH — Volume f64→i64 silent overflow

**Files:** `trading.rs:3134-3140` (`units_to_ctrader_protocol_volume`)
and `trading.rs:3142-3147` (`ctrader_protocol_volume_from_lots`).
Both do `(value * factor).round() as i64` with no bounds check. f64
values larger than `i64::MAX / factor` overflow to `i64::MIN` (or
`i64::MAX` saturating, depending on platform — Rust 2021 says it's
saturating cast, but the round() result becoming +∞ is UB-adjacent).
A buggy strategy producing `volume = 1e18` will silently submit a
broker order with arbitrary size.

**Fix:** Guard with `if (value * factor).abs() > i64::MAX as f64 { return Err(…) }`.

---

#### F-MODELS6-001 CRITICAL — OPA `tau` no finite check before weight update

**File:** `crates/forex-models/src/streaming/adaptive_impl.rs:695`
```rust
let tau = (margin * sample_weights[*target_class] / (2.0 * norm_sq))
    .min(self.aggressiveness);
// ... weight update applied with tau
```

Pass 2 flagged the missing finite check (F-MODELS3-013) as MEDIUM. Pass 3
reading shows it's actually CRITICAL: `.min()` on NaN returns NaN
(`NaN.min(x) == NaN`), so a single non-finite intermediate
silently corrupts the entire weight matrix, and the streaming learner
will produce non-finite predictions thereafter. Online learners can't
recover from poisoned weights without restart.

**Fix:** Add `if !tau.is_finite() || tau < 0.0 { continue; }` immediately
after the calculation.

---

#### F-MODELS6-002 HIGH — Hoeffding fallback gradient no finite check on error

**File:** `adaptive_impl.rs:933` — same class of bug as F-MODELS6-001
but in the SGD fallback for the Hoeffding committee. NaN in `error`
poisons all subsequent weight updates. No margin-based safeguard
(unlike OPA), so this path is even more fragile.

#### F-MODELS9-001 CRITICAL — DQN tests assert hardcoded values without proof of derivation

**File:** `crates/forex-models/src/rl/dqn_impl_tests.rs:95-102`
```rust
assert!((q_values[0] - 0.35).abs() < 1e-6);
assert!((q_values[1] - 0.95).abs() < 1e-6);
assert!((q_values[2] - 0.4).abs() < 1e-6);
```

Hardcoded expected values with no in-test derivation comment. The
expected numbers happen to match `weights @ input + bias` for the
test setup (`[1,0;0,1;0.5,0.5] @ [0.25,0.75] + [0.1,0.2,-0.1]`),
but the test does not encode that formula in code — it just asserts
numbers. If the production code starts using a different forward
pass (e.g., applies a non-linear activation), the test could still
pass if someone updates the hardcoded numbers to match the new buggy
behavior. **Tautology test that doesn't gate the contract.**

**Fix:** compute expected values inline from the input weights/bias
in the test body, and assert against the computed value, not magic
numbers.

---

#### F-MODELS9-004 HIGH — DQN tests pre-bake "good" reward shape

**File:** `dqn_impl_tests.rs:35-113` — Test sets
`average_buy_reward = 0.2 > average_hold_reward = 0.1 > average_sell_reward = -0.1`
manually in the metadata, then asserts the saved/loaded artifact
preserves these magnitudes. But the **production** reward computation
that should produce these values is never exercised. Tests that
"buy is best in this synthetic uptrend" only check the metadata
serialization, not whether the agent actually learned correctly.

**Fix:** Generate the synthetic environment, run the production
reward function on it, capture the actual rewards, then assert the
saved artifact matches.

**2026-05-15 follow-up:** because the operator directive forbids
inventing synthetic broker data inside tests, the existing test was
renamed from `fallback_only_artifact_round_trips_without_network_file`
to `fallback_only_artifact_serde_roundtrip` and a leading comment
was added documenting that it only validates serde + the fallback
inference algebra, NOT the correctness of the metadata reward
magnitudes. The aspirational learning test that would actually
exercise `build_reward_triplet` end-to-end is now registered as
`trained_agent_learns_action_reward_ordering_on_real_data` with
`#[ignore = "needs real cTrader-historical fixture and a full
training pass; tracks F-MODELS9-004"]`, and continues to track this
finding until a real cTrader historical fixture and a deterministic
full-training harness are available.

### Cluster B — Numerical / GPU correctness in evolutionary code

Pass 3 read crfmnes_impl, neat_impl, forest_impl in full and found
~30 medium-severity numerical issues:

- **F-MODELS7-001 MEDIUM** — Box-Muller transform clamps `u1` to
  `f64::MIN_POSITIVE` (≈ 2.2e-308), then takes `ln` → produces values
  near -708 → `sqrt(-2 * -708) ≈ 37.6` → outlier samples 30+ standard
  deviations from the mean, polluting CRFMNES covariance adaptation.
  Fix: clamp to `[1e-10, 1.0 - 1e-10]`.
- **F-MODELS7-002 MEDIUM** — CRFMNES feeds `f64::INFINITY` to the
  optimizer's `tell()` for non-finite raw losses, while a separate
  external `best_loss` tracker only sees finite values. The two
  trajectories diverge across generations.
- **F-MODELS7-009 MEDIUM** — Multi-island training with mixed GPU/CPU
  fitness: GPU fitness uses f32, CPU uses f64. If GPU disables
  mid-run (line 1021), island N+1 produces systematically different
  loss values than island N, biasing best-of-island selection.
- **F-MODELS7-010 MEDIUM** — `best_params` stored as f32 (after f64→f32
  conversion for GPU eval) but `best_selection_loss` stored as f64.
  Re-evaluating `best_params` after fit will produce different loss
  than recorded.
- **F-MODELS7-019 MEDIUM** — CRFMNES seeds from `rand::random()` per
  island → **non-deterministic re-fits**. Same input, same config →
  different output. Save/load is reproducible, but `fit()` is not.
- **F-MODELS8-005 MEDIUM** — NEAT `break_cycles()` silently mutates
  topology after mutation, so the fitness measured includes an
  unaccounted complexity adjustment.
- **F-MODELS8-007 MEDIUM** — `runtime_backend.contains("cuda")` matches
  any backend label containing "cuda" (e.g., "cuda_fake", "scuda").
  Should be exact match against an enum.
- **F-MODELS8-015 MEDIUM** — Isolation forest quantile uses `.round()`
  index, not linear interpolation. 95th percentile on 10 samples
  resolves to index 9 (the max), pushing the threshold to the
  extreme value → false-positive anomaly rate inflates.
- **F-MODELS8-018 MEDIUM** — Extended isolation forest dispatch uses
  `seq!(N in 1..=128)` macro, hard-limited to 128 features. Datasets
  with 129+ features silently fall back to the diagonal profile, with
  no fit-time warning (only logged at load time).

### Cluster C — Adaptive learner edge cases

Pass 3 read all 2178 lines of `adaptive_impl.rs` and found 17 findings.
Highlights beyond Cluster A:

- **F-MODELS6-004 MEDIUM** — `balanced_class_weights` uses `count.max(1)`
  for missing classes → inflates weight for unobserved class →
  unstable updates if that class is later predicted.
- **F-MODELS6-007 MEDIUM** — `aggressiveness.max(1e-4)` allows arbitrarily
  large values; should be `.clamp(1e-4, 100.0)`.
- **F-MODELS6-008 MEDIUM** — Drift detector params (`drift_delta`,
  `warning_level`, etc.) passed to `irithyll` crate without bounds
  validation; invalid values may produce undefined behavior.
- **F-MODELS6-009 MEDIUM** — Hoeffding committee restoration logic has
  a redundant check that masks a real gap: if `committee_json` is
  empty and `failures > 0`, no error is raised before
  `committee_runtime_ready` is set based on payload presence alone.
- **F-MODELS6-010 MEDIUM** — `total_cmp` for argmax on NaN-poisoned
  scores: NaN ranks as Less, so the predicted class is arbitrary
  if any score is NaN. Combined with F-MODELS6-001, NaN can flow
  through the entire predict path silently.
- **F-MODELS6-011 MEDIUM** — Quadratic feature expansion has no
  dimension sanity check; 1000 input features → 2000 expanded
  features → unbounded matrix sizes.

### Cluster D — Test-quality issues (DQN + swarm)

Pass 3 read both god-test files (1492 + 1370 LOC) and found 15 findings
about **what the tests don't actually verify**. Highlights:

- **F-MODELS9-002 HIGH** — `validate_q_values_rejects_non_finite_rows`
  only tests NaN. Missing: `f32::INFINITY`, `f32::NEG_INFINITY`, mixed
  finite+infinite, all zeros, all-equal.
- **F-MODELS9-003 HIGH** — Backend drift test asserts substring of
  degraded reason, but doesn't verify multiple drifts (device_policy
  also differs in the test setup) are all reported.
- **F-MODELS9-005 MEDIUM** — Feature scaler test hardcodes
  `expected = [2.0, 2.0]`. If implementation switched from z-score
  to min-max scaling and the hardcoded numbers happened to match
  by coincidence, test would still pass.
- **F-MODELS9-006 MEDIUM** — Device policy normalization tests miss:
  case sensitivity (`"cuda:0"` lower vs `"CUDA:0"` upper),
  invalid indices (`"cuda:99"`), missing indices (`"cuda:"`),
  whitespace handling.
- **F-MODELS9-007 HIGH** — Swarm forecaster save/load roundtrip:
  asserts field-by-field but doesn't verify aggregate invariants
  (weights sum to 1.0 after load, count of reports preserved,
  ordering preserved).
- **F-MODELS9-008 HIGH** — `learned_validation_weights_favor_lower_loss_candidates`
  only checks ordering (`best > mid > worst`), not magnitude or
  edge cases (equal losses → equal weights, perfect prediction →
  near-100% weight, missing candidates → error).
- **F-MODELS9-013 HIGH** — Stale-artifact rebuild test verifies
  `fitted_horizon == 12` (the new horizon) but doesn't test what
  happens when the new horizon is incompatible with available
  data (e.g., 8 data points, asking for 12-step forecast).

These are **test quality** issues — the tests pass, but they don't
actually constrain the production code's behavior. They're listed
separately because fixing them won't change runtime behavior; they'll
just close the gap that lets refactors silently break invariants.

## What this audit can confirm and what it can't

### What it can confirm

- Every `.rs` file in `crates/` has been read end-to-end at least once
  across the three passes.
- Every CRITICAL/HIGH finding cites a specific `(file, line range,
  evidence excerpt)` that an agent in this session read directly.
- The audit is no longer extrapolating from prior (deleted) audit docs.
- The historical worry that the audit was reading docs instead of code
  is now resolved.

### What it still can't confirm

- **Runtime behavior**: This is a static audit. We have not run
  `cargo build`, `cargo clippy`, `cargo test`, or any of the actual
  trading subsystems. The 3 conflicts identified in Pass 2
  (F-SEARCH-002 vs F-SEARCH2-005, the ML lookahead findings, the
  trading.rs god-class) require runtime instrumentation to resolve
  definitively.
- **Cross-file emergent behavior**: Each agent saw a slice of the
  codebase. Bugs that only manifest from interaction between
  subsystems (e.g., `forex-data` produces a value that
  `forex-models` consumes incorrectly) may still exist beyond what
  the per-crate audits caught.
- **Concurrency under real load**: The atomic-ordering and
  race-condition findings (F-APP3-001, the env::set_var findings
  from Pass 2, etc.) are reasoned from code patterns. Whether they
  manifest as actual data corruption depends on scheduler behavior
  that only shows up under VPS load.
- **Whether each finding is exploitable**: We haven't written
  proof-of-concept exploits for the security findings (OAuth
  state CSRF, client_secret in URL, plaintext API key). They are
  real attack surfaces but their exploitability depends on
  deployment posture.
