### 5.2 Two production gaps the removal SURFACED — both FIXED

The `TradingSession` was the (dead) integration hub where several features had
been wired, so they never ran in production. Two were re-wired to the live path:

1. **`journal_reconcile` (task #17) was never populating the journal.** Its only
   caller was the dead `TradingSession` heartbeat (`trading/mod.rs:1915`).
   **Fix:** call `reconcile_best_effort(&snapshot)` from the production
   account-refresh path (`server/bridge.rs`, after `load_account_runtime`),
   fire-and-forget on the blocking pool. The journal now fills on every
   dashboard poll (idempotent on `position_id`).
2. **OAuth token never auto-refreshed in production.** `bridge.rs::refresh_once`
   loaded the token as-is; after the TTL boundary every account fetch broke
   until a manual browser re-auth. The silent-refresh impl
   (`ProductionCTraderLiveAuthBackend::refresh_token_bundle`) existed but was
   unwired. **Fix:** new `refresh_ctrader_token_if_needed()` helper in
   `bridge.rs` — best-effort, non-fatal (on any failure it keeps the existing
   token → no regression), 30-min refresh-ahead window, persists to the keyring.

### 5.3 Audit finding — production manual orders are ungated (BY DESIGN)

> **Operator decision (2026-06-03): manual trading stays manual — leave the
> `/orders` path ungated.** A manual order is a deliberate operator action;
> blocking it with an automated risk gate would be wrong. The finding below is
> recorded for accuracy, but it is NOT a gap to fix on the manual path. Risk
> enforcement is relevant only for AUTONOMOUS order placement (see §5.5).


A 3-agent read-only audit verified (and I confirmed by hand) that the production
order path enforces **no risk limits**:

> `POST /orders` → `orders::place` → `submit_market_order_blocking`
> (`broker_api.rs:657-823`) → `ProductionCTraderExecutionBackend::execute` →
> live cTrader. The only pre-send guards are mechanical (volume/SL/TP finite).
> **No** risky-mode arming check, **no** prop-firm drawdown/position limit,
> **no** kill-switch.

- Pre-existing — the gate lived in the dead `TradingSession` (`trading/orders.rs`
  called `prop_firm_pre_trade_check` + `record_kill_switch_trip`), which never
  ran. The enforcement engine **exists and is unit-tested** in core
  (`RiskyModeManager::check_trade_allowed` `risky_mode.rs:683`,
  `RiskManager::check_trade_allowed` `risk.rs:505`) — it is simply never called
  from the app's order path.
- The risky-mode **kill-switch is inert**: `record_kill_switch_trip` has zero
  callers (nothing trips it); nothing reads `armed` before a trade; the only
  live kill-switch code is the 24 h auto-RE-arm in `bridge.rs` — it re-arms a
  switch nothing ever trips and nothing ever reads.
- **NOT a manual-path gap** (operator decision above). It matters only when
  AUTONOMOUS live trading is built — that path must carry the gate (§5.5).
