# NeoEthos Autonomous Trader — Design (v0.5)

> Status: **DESIGN** (nothing built yet). This is the blueprint for the single
> most important remaining build: turning NeoEthos from a *research lab + manual
> terminal* into a bot that trades the user's discovered strategies / trained
> ensemble **on its own, across every trained pair and timeframe, independent of
> what the user is looking at on screen.**

---

## 0. Why this exists (the gap, verified 2026-06-04)

A full audit of the live binary confirmed: **discovery + training produce
strategy/model artifacts that NOTHING consumes for live trading.** The two halves
exist and are largely solid:

- **Brain** (ML): discovery (`neoethos-search`), training (`TrainingOrchestrator`),
  and a real inference engine (`neoethos-models/src/ensemble_inference/` —
  `SoftVotingEnsemble`, `ExpertRegistry::load_with_partial`).
- **Hands** (execution): `ProductionCTraderExecutionBackend`
  (`ctrader_execution.rs`) + `broker_api.rs` — market open / close / **partial
  close** / cancel are wired and functional.

…but the **nervous system that connects them is absent** — it was deleted with
the old `app_services::trading` (`auto_trade`, `auto_trade_producer`,
`ensemble_predictor_adapter`, `risk_gate`; see `trading_types.rs:5-13`) and never
rebuilt. Concretely missing:

| Missing piece | Evidence |
|---|---|
| Autonomous decision loop | `main.rs:415-444` spawns only bridge + tick-cache + HTTP; pipeline auto-chains Discovery→Training and **stops** (`engines_control.rs:325-328`, “Auto-Trader — lands in a follow-up”) |
| Live multi-TF feature pipeline | feed streams **ticks only** for ~8 UI symbols (`live_spots_streamer.rs:74-76,503-515`); no bar aggregation, no live feature cube |
| Data feed driven by trained pairs | subscribes the **watchlist/majors**, not the portfolio (`live_spots_streamer.rs:191-204`) |
| Ensemble → order wiring | `ensemble_inference` has **zero consumers** in `neoethos-app`; `live_models/` is written by promotion and **read by nobody** |
| Risk gate on execution | `RiskyModeManager`/`RiskManager::check_trade_allowed` exist in core but are **not called** on any order path |
| Modify SL/TP (trailing) | `build_amend_order_request` exists but no `CTraderExecutionRequest::AmendOrder` variant / `amend_order_blocking` |
| Limit/stop orders | `submit_market_order_blocking` hardcodes `Market` |
| CLI GPU | `neoethos-cli` has no `gpu-*` feature → VPS deep-search is CPU-only |

**Net today:** the bot shows a chart, runs research, and lets *you* click
Buy/Sell. It cannot trade your ensemble by itself.

## 1. Goal

A background service that, **independent of the UI**:

1. Loads the **promoted portfolio** (which pairs + TFs + strategies/models are live).
2. Watches **every** trained pair in **real time**, aggregating the TFs each
   strategy trades.
3. Computes the **same engineered features** discovery/training used, live.
4. Runs the **strategy logic and/or the trained ensemble** to produce signals.
5. **Opens / closes / partially closes / trails** positions through the broker.
6. Is **risk-gated** (prop-firm rules in PropFirm mode; equity-floor + kill
   switches in Risky mode) and **safe by default** (paper-trade first, opt-in live).

The user's two target regimes (already modelled in config):
- **PropFirm**: pass the challenge with **no rule breaches** + **consistent
  monthly** profit.
- **Risky**: grow €100 → €100k, **without losing the initial principal** (hard
  equity-floor — see §8).

---

## 1.1 Parity mandate (NON-NEGOTIABLE)

**Discovery, training, and the autonomous trader MUST behave IDENTICALLY no
matter how they are launched — UI/backend HTTP, CLI, or headless. No capability,
default, or resolved config may differ between front-ends.** A UI↔CLI divergence
is a bug, not a "mode".

- **GPU everywhere.** `neoethos-cli` now mirrors `neoethos-app`'s GPU features
  exactly (`gpu-vulkan` / `gpu-nvidia` / `gpu-rocm`); the CLI/VPS/headless path
  runs the SAME cubecl GPU + CPU/iGPU-hybrid eval as the UI — never silently
  CPU-only. (Fixed; was the root of the "VPS is CPU-only" symptom.)
- **One config resolution.** Higher-TF ladder, discovery knobs, mode, and risk
  policy resolve through ONE shared path (`canonical_higher_timeframes`,
  `DiscoveryConfig::from_settings`) so the UI and CLI compute byte-identical
  effective config for the same settings. (Open: a couple of legacy config
  mechanisms — `multi_resolution_timeframes` vs `higher_timeframes` — still
  diverge; see the parity-audit task.)
- **One engine.** The autonomous trader is a SINGLE library service (core/app
  crate); the UI and CLI are thin front-ends that only start/stop/inspect it.
  There is no UI-only or CLI-only trading logic, ever.

A feature is "done" only when it is reachable AND identical from both
front-ends.

## 2. Architecture

A single supervised `tokio` task tree, `autonomous_trader`, spawned at backend
startup **behind a config flag** (`autonomous.enabled`, default `false`) and a
**mode** (`Paper` | `Live`).

```
                ┌─────────────────────── AutonomousTrader (supervisor) ──────────────────────┐
                │                                                                             │
 model_targets  │  PortfolioRegistry ──▶ active set: [(symbol, base_tf, higher_tfs, source)] │
 live_models/   │        │ (hot-reload on promotion)                                         │
                │        ▼                                                                    │
 cTrader WSS ──▶│  LiveMarketData ──▶ per (symbol,tf) rolling OHLCV buffers (bar close evts) │
                │        │                                                                    │
                │        ▼ on bar-close                                                       │
                │  LiveFeaturePipeline ──▶ multi-TF feature row (same cube as training)       │
                │        │                                                                    │
                │        ▼                                                                    │
                │  SignalEngine ──▶ Signal{dir, confidence, source}   (strategy ⊕ ensemble)  │
                │        │                                                                    │
                │        ▼                                                                    │
                │  DecisionEngine ──▶ TradeIntent{Open|Close|Partial|Amend, vol, sl, tp}      │
                │        │                                                                    │
                │        ▼                                                                    │
                │  RiskGate (check_trade_allowed) ──▶ allow / reject(tier)                    │
                │        │ allow                                                              │
                │        ▼                                                                    │
                │  ExecutionAdapter  ── Paper(sim fills) | Live(broker_api)                   │
                │        │                                                                    │
                │        ▼                                                                    │
                │  PositionManager ──▶ open positions, trailing, partial TP, exposure caps    │
                └─────────────────────────────────────────────────────────────────────────────┘
```

### Components & responsibilities

- **PortfolioRegistry** — reads `cache/discovery/*.json` portfolios + promoted
  `live_models/<symbol>/<tf>/` + `model_targets.json` → the active
  `(symbol, base_tf, higher_tfs, StrategySource)` list. Hot-reloads when
  `strategy_lab::promote_if_gated` writes a new promotion. *This is the source of
  truth for "what to watch" — NOT the watchlist.*
- **LiveMarketData** — subscribes the cTrader feed to **the registry's symbols**;
  aggregates ticks → bars per required TF; emits a `BarClosed{symbol,tf,ohlcv}`
  event. Maintains a rolling window long enough for the longest indicator warmup.
  (Extends `live_spots_streamer`; adds trendbar subscription /
  `ProtoOAGetTrendbarsReq` backfill + tick→bar aggregation.)
- **LiveFeaturePipeline** — on each `BarClosed`, builds the **same** multi-TF
  feature row used in discovery (`compute_hpc_feature_frame` +
  `prepare_multitimeframe_features` semantics) for `(symbol, base_tf + higher_tfs)`.
  Incremental/rolling where possible; windowed-recompute as the simple first cut.
- **SignalEngine** — per `(symbol, base_tf)`:
  - *Strategy path*: run the discovered `Gene` exactly as the GA did
    (`synthesize_signals_cpu` for the latest bar) → Long/Short/Flat.
  - *Ensemble path*: run `SoftVotingEnsemble` on the feature row → class vote +
    confidence.
  - *Blend*: regime-conditional gate (see §7) — ensemble decides *whether to
    trust* the strategy in the current regime.
- **DecisionEngine** — maps a signal + current positions to a `TradeIntent`
  (open with the strategy's SL/TP, close on exit signal, scale-out partial at
  R-multiples, amend SL for trailing). Sizing: PropFirm → `RiskManager`
  fixed-fractional; Risky → `RiskyModeManager::calculate_position_size_usd`.
- **RiskGate** — every intent passes `RiskManager::check_trade_allowed` (PropFirm:
  daily-loss, max-DD, drawdown-recovery tiers) or
  `RiskyModeManager::check_trade_allowed` (kill switches, daily/monthly caps,
  **equity-floor §8**) before execution. Reject → drop + log.
- **ExecutionAdapter** — ONE real cTrader path
  (`broker_api::{submit_market_order_blocking, close_position_blocking,
  cancel_order_blocking}` + new `amend_order_blocking`). **DEMO vs LIVE is purely
  the connected broker ACCOUNT (fake vs real money), NOT separate bot code** — the
  data, features, models, and signals are byte-identical either way; only where
  the fills land differs, and cTrader already provides demo accounts. (A
  trait-object boundary exists only so unit tests can inject a mock, and an
  optional `--replay` runs the loop over historical bars for fast offline
  validation. Neither is a user-facing "mode" — there are NOT two buttons for the
  same thing.)
- **PositionManager** — tracks open positions per symbol (and per strategy),
  enforces max-concurrent-positions + portfolio exposure caps, drives trailing
  stops (via Amend) and partial take-profit, attributes P&L per strategy (feeds
  the journal + the model-training corpus).

## 3. The loop (per bar-close, per pair)

```text
on BarClosed(symbol, tf):
    if tf is a base_tf in registry:
        feats   = LiveFeaturePipeline.row(symbol, tf)          # multi-TF cube
        signal  = SignalEngine.evaluate(symbol, tf, feats)     # strategy ⊕ ensemble
        intent  = DecisionEngine.intent(symbol, signal, PositionManager.open(symbol))
        if intent.is_some():
            match RiskGate.check(intent):
                Ok       => ExecutionAdapter.execute(intent);
                            PositionManager.apply(intent)
                Reject(t)=> log_blocked(intent, t)
    # always, every tick or short timer:
    PositionManager.manage(symbol)   # SL/TP hit, trailing amend, time-stop, partial TP
```

All pairs run concurrently (one task per symbol, or a shared work-stealing pool).
**Nothing here reads the UI state** — the UI is a *viewer* of the same data.

## 4. Build phasing (each phase ships + is verifiable on its own)

- **Phase 1 — Skeleton + offline replay (safe).** Supervisor task +
  `PortfolioRegistry` + a mock ExecutionAdapter / `--replay` over historical bars
  + `PositionManager` + a `/autonomous/*` status API. Wire a *stub* SignalEngine
  (the strategy gene path on replayed bars). Proves the loop end-to-end with
  **zero broker calls** — a pure dry-run, not a parallel "paper" product.
- **Phase 2 — Live multi-TF data.** Extend `live_spots_streamer` to subscribe the
  registry's symbols + trendbar backfill + tick→bar aggregation → `BarClosed`.
- **Phase 3 — Live feature pipeline.** The rolling multi-TF feature cube per pair
  (the biggest lift; reuse `compute_hpc_feature_frame`). Validate live features
  match a batch recompute on the same bars (parity test).
- **Phase 4 — SignalEngine.** Wire the strategy `Gene` path + the
  `SoftVotingEnsemble` path; the regime-conditional blend (§7). Paper-trade it for
  weeks; compare paper P&L vs the strategy's backtest expectancy.
- **Phase 5 — RiskGate + Live execution.** Wire `check_trade_allowed`; add
  `CTraderExecutionRequest::AmendOrder` + `amend_order_blocking`; flip
  `LiveExecutionAdapter` on with **tiny** size + a hard kill switch.
- **Phase 6 — Position/exposure mgmt + scale.** Trailing, partial TP ladders,
  max-concurrent + exposure caps, all trained pairs concurrently, a live
  monitoring dashboard + per-strategy attribution.

**Gate before LIVE money:** N weeks on a cTrader **DEMO account** (real broker,
real market, fake money — the SAME bot/data/models/signals as live) where the
demo P&L tracks backtest expectancy within tolerance, zero rule breaches, and
only OOS-tagged strategies (surfaced in `promotion_summary.json`) are promoted.
"Going live" = connecting a live account; **nothing else in the bot changes.**

## 6. Key contracts (sketch)

```rust
struct PortfolioEntry { symbol: String, base_tf: String, higher_tfs: Vec<String>,
                        source: StrategySource, mode: TradeMode }
enum StrategySource { Gene(Gene), Ensemble { dir: PathBuf }, Blend(Box<Gene>, PathBuf) }
enum TradeMode { PropFirm, Risky }

struct LiveBar { symbol: String, tf: String, o: f64, h: f64, l: f64, c: f64, ts: i64 }
enum Direction { Long, Short, Flat }
struct Signal { symbol: String, dir: Direction, confidence: f64, source: SignalSource }

enum TradeIntent { Open { sym, dir, vol, sl, tp }, Close { pos_id, vol: Option<f64> },
                   Amend { pos_id, new_sl, new_tp }, Cancel { order_id } }

trait ExecutionAdapter { fn execute(&self, i: TradeIntent) -> Result<ExecReport>; }
trait RiskGate { fn check(&self, i: &TradeIntent, snap: &AccountSnapshot) -> Result<(), KillSwitchTier>; }
```

## 7. Model-feeding loop (the user's vision, integrated)

The discovered strategies are **short-lived** (no edge survives 8 years) — so the
robustness comes from a **diverse, OOS-tagged library** feeding a
**regime-conditional meta-model**, not from one eternal strategy:

1. Discovery produces many strategies; each is **tagged** with its held-out OOS
   reliability (forward-test/prop-firm verdict — *now surfaced in
   `promotion_summary.json`*).
2. Training feeds this tagged corpus (+ regime labels, cross-strategy correlation,
   failure cases, news/macro context) into the ensemble.
3. The ensemble becomes a **gating network**: given the *current live regime*
   (trend/range/vol/session/news), it decides **which strategies to trust now**.
4. The autonomous loop runs that regime-conditional ensemble across all pairs →
   activates the right strategies for the moment, sizes them, manages risk.

This is what makes "all the pairs/TFs it was trained on" actually pay off: the bot
isn't running one strategy everywhere — it's running the **right** strategies per
pair per regime.

## 8. Safety & risk policy

- **Default OFF.** `autonomous.enabled=false`; explicit operator opt-in + signed
  AutonomyRisk acknowledgement (the `autonomous_only_contract_accepted` gate
  already in `RiskyModeManager::new`).
- **Demo account first.** Validate on a cTrader DEMO account (fake money, real
  broker) until the demo-P&L-vs-backtest gate passes; going live only swaps the
  connected account. There is **no separate paper-simulation mode** — demo vs
  live is purely which broker account is connected; the bot, data, models, and
  signals are identical.
- **PropFirm mode** → `RiskManager` enforces daily-loss / max-DD / consistency;
  only OOS-passing strategies promoted; target = pass the challenge, monthly
  consistency.
- **Risky mode → hard equity-floor (the "don't lose the principal" contract).**
  Add `initial_bankroll_usd` field + `principal_floor_fraction` config +
  `KillSwitchTier::PrincipalFloor`; `check_trade_allowed` halts (naturally sticky,
  bankroll-gated) when `current_bankroll < initial * principal_floor_fraction`.
  Combined with the existing per-stage retreat + taper, this caps catastrophic
  loss while still risking *gains* (house money) to compound.
- **Manual HALT** kill switch always live; hardware/connection-loss flatten.
- **Risk gate is mandatory on the autonomous path** (it's optional/by-design-off
  for manual UI orders only).

## 9. Open decisions (need operator input before Phase 4)

1. **Signal source:** strategy-gene only / ensemble only / **blend** (ensemble as
   a regime gate over genes)? — recommend blend.
2. **Live features:** rolling-incremental (fast, complex) vs windowed-recompute
   (simple, heavier) for the first cut? — recommend windowed first, optimise later.
3. **Multi-strategy-per-pair:** net signals into one position, or one position per
   strategy (cleaner attribution, more positions)? — recommend net + attribute.
4. **Sizing:** per-strategy fixed-fractional vs portfolio-level (quarter-Kelly)?
5. **Paper duration** before any live money?
6. **Pair universe at launch:** all promoted pairs at once, or roll out a few?

## 10. Non-goals (for v0.5)

- No new broker integration (cTrader only).
- No HFT / sub-second logic (bar-close cadence).
- No discretionary override mixing (autonomous and manual stay separate paths).
