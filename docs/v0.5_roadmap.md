### 2.4 f32 conversion (delayed for operator decision)

- **Scope:** Agent C catalogued **1913 conversion sites** in the
  Pass 1 audit (commit `7000c1e` adds the catalogue under
  `forex-models/src/`). Pure-math sites (geometry, RNG, sampling)
  are unambiguous → f32. Financial domain types (lot sizes, prices,
  PnL, margin) sit in the **AMBIGUOUS bucket** pending a
  precision-tradeoff decision: f32 saves memory and matches GPU
  defaults (per `ml_numerical_reference.md` §1 — burn defaults to
  f32 everywhere) but loses ~7 digits of mantissa precision,
  which for cash values denominated in USD millions is one digit
  too few.
- **Severity:** P0 (operator-decision-gated; can't ship v0.5
  without a verdict because too much downstream code branches
  on the answer).
- **Effort:** 1 wk if the verdict is "f32 everywhere with
  scaled-integer cents at the boundary"; 3 wk if "f64 only for
  domain types, f32 for everything else" because of the
  boundary-conversion lattice.
- **Dependencies:** operator decision — see §7.

## §6 — Risk register

Honest enumeration of v0.4.5 / v0.5.0 release risks. Mitigations
are tracked.

### 6.1 — 5 agents in flight may overlap on chrome / orders.rs

- **Risk:** The five concurrent v0.4.5 work-streams (HALT button,
  status pill, wizard scaffold, moneyDigits propagation, UnrealizedPnL
  final wire) all touch `crates/forex-app/src/ui/` and
  `crates/forex-app/src/app_services/trading/orders.rs`. Despite
  prompt-level coordination ("HALT button goes in
  `chrome/status_bar.rs`, never edit `orders.rs`"), an agent can
  silently extend the touch-point.
- **Severity:** HIGH for v0.4.5.
- **Mitigation:** Mandatory rebase to HEAD before each agent's
  final commit; reviewer runs `git diff --name-only HEAD~1 HEAD`
  on every WIP commit to catch path overlap early. The `orders.rs`
  + `mod.rs` decomposition complete in `26873b0` already narrows
  the conflict surface vs the previous 3372-LOC `trading.rs`.

### 6.2 — catboost-rust sandbox failure blocks `cargo check
--all-targets`

- **Risk:** The default-feature build cannot be validated in the
  pre-VPS sandbox because catboost-rust's `build.rs` fetches C
  headers over network and the sandbox produces `UnknownIssuer`
  TLS errors. Result: no full workspace check until VPS.
- **Severity:** MEDIUM. The `--no-default-features --features
  tree-models-core,pure-rust-ml` path does check clean (14s +
  1m50s for GPU features), so the ~98 % majority of the workspace
  is validated locally; only the catboost-rust integration is
  unverified.
- **Mitigation:** v0.4.5 gate (§5.1 item 2) explicitly accepts
  the no-default-features build matrix as the canonical check.
  VPS smoke-test reruns the default-feature build before v0.5.0.

### 6.3 — Homebrew Cask 225-star notability rule

- **Risk:** Homebrew Cask self-submitted casks need ≥ 225 stars on
  the source repo per
  `installer_no_paid_certs_strategy.md` §2.2 (Acceptable-Casks.md
  citation). `kosred/forex-ai` is currently private; even after a
  public release, hitting 225 stars takes time.
- **Severity:** LOW (no v0.4.5 impact; v0.5.0 affected if Cask is
  a hard release-bar, which it is not).
- **Mitigation:** ship the unsigned `.dmg` directly from GitHub
  Releases with the user-side `xattr -dr com.apple.quarantine`
  README per `installer_no_paid_certs_strategy.md` §2.4 (Path C).
  Run an operator-hosted Homebrew tap (`brew tap kosred/forex-ai
  && brew install --cask forex-ai`) as the v0.5 interim path; PR
  to upstream `Homebrew/homebrew-cask` only once the star
  threshold is met.

### 6.4 — Spotware proto refresh introduces silent payload changes

- **Risk:** The upstream `OpenApiMessages.proto` grew +15.5 KB
  (35 KB → 50.6 KB) between our 2026-05-08 snapshot and the
  2026-05-14 refresh per `spotware_proto_freshness.md` §1. Pure
  additions (+30 new messages), so backward-compatible at the wire
  level — but if upstream adds an `optional` field to an existing
  message and we don't pick up the `[default = …]` annotation,
  serialisation can diverge.
- **Severity:** MEDIUM.
- **Mitigation:** the Pass 3 `26873b0` Batch 13 integration tests
  cover the full round-trip on Group A/D/E/F/G. Group B/C are not
  wired so the new messages can't yet desync. Pin a known-good
  upstream commit hash in a comment block at the top of each
  local proto file per `spotware_proto_freshness.md` Next-Steps
  item 3.

- **Risk:** The four operator decisions in §7 each block a v0.5

## §7 — Sign-off check

Four operator decisions gate v0.4.5 → v0.5.0:

### 7.1 Acceptable Risky Mode initial-stage ruin probability

Per `risky_mode_compounding_research.md` §10.1, suggested ceiling is
**50 %** for the S1 stage. Background: §1.3 of the research puts the
canonical 20-pip-challenge ruin probability at "95 %+" from anecdotal
retail sources. Halving that to 50 % is a defensible, honest win.
30 % collapses S1 into "FTMO Standard with a different name"; 70 %
loses the "safer" framing. Operator range: [10 %, 70 %].

**Decision needed:** ceiling value.

### 7.2 f32 precision tradeoff in financial-domain types

Agent C's audit catalogued 1913 conversion sites; the ambiguous bucket
is the financial-domain types (lot sizes, prices, PnL, margin) where
f32's ~7 digits of mantissa precision is one digit short for USD
millions. Two options:

- **Option A — f32 everywhere with scaled-integer cents at the
  boundary.** Match `ml_numerical_reference.md` §1 guidance (burn
  defaults to f32). Effort: 1 wk.
- **Option B — f64 for domain types, f32 for everything else.**
  Boundary-conversion lattice across crates. Effort: 3 wk.

**Decision needed:** Option A or Option B.

### 7.3 cTrader .NET SDK adoption (Yes / No / Defer)

Per `ctrader_api_reference.md` §5, the SDK is MIT-licensed, last
published 2022-05-03, .NET 6 target (out of LTS). Research
recommendation: **stay on Rust + prost** because the SDK is
maintenance-mode and may already lag the live wire format. Operator
raised the question; the deliverable answers it.

**Decision needed:** Yes (migrate) / No (stay on Rust) / Defer
(re-evaluate at v0.6).

### 7.4 Real-data fixture capture strategy

Per §2.6, 21 sites are marked `TODO(real-data)`. Two capture
strategies:

- **Operator-supplied** — operator runs a one-shot capture session
  on their live cTrader demo account and exports the proto-binary
  payloads. Privacy-clean (operator's account, operator's data).
- **Public demo-account capture** — operator (or maintainer) opens
  a fresh demo account on a public cTrader broker, runs the same
  capture, exports. Privacy-clean (synthetic-but-broker-real data).

**Decision needed:** operator-supplied / public demo-capture.
