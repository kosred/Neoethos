# v0.5 Roadmap — Distribution-ready first beta

**Owner:** kosred
**Started:** 2026-05-14
**Base release:** v0.4.0 (commit `0091308`)
**Target release:** v0.5.0 — first version handed to external beta testers
**Strategy:** incremental patch releases `v0.4.1 → v0.4.2 → … → v0.4.N`,
each one shippable on its own. Every patch must leave `master` in a state
where `forex-cli` and `forex-app` both run and produce sensible output.

---

## 1. Vision for v0.5

A user who has **never seen this repo before** must be able to:

1. Download a single archive for their OS (Windows or Linux).
2. Run **one** command (or double-click an installer / binary).
3. See the app launch, auto-detect or auto-import data, and produce
   a non-empty strategy portfolio on at least one major pair (EURUSD)
   **without editing any config file, exporting any env var, or
   reading a README first**.
4. Reach a working chart + journal in the GUI within 60 seconds.
5. Hit "Start" on Discovery / Training without touching JSON.

Anything that requires the user to read source code, copy env vars, or
debug a stack trace counts as a **v0.5 blocker** and must land in one of
the v0.4.x patches below.

## 2. Known gaps (audit-grounded)

These are the concrete, identified blockers as of v0.4.0. Each one maps
to a patch milestone in §3.

### 2.1 Discovery yield — empty portfolios on EURJPY / XAUUSD

- **Symptom:** default-config discovery on EURJPY and XAUUSD across
  M30 / H1 / H4 / D1 emits 0 strategies. EURUSD / GBPUSD / AUDUSD work.
- **Root cause narrowed:** NaN propagation in the GA's combined signal
  under raw feature magnitudes (1e8–1e11 range on JPY pairs + metals).
- **What did NOT fix it:** opt-in median/MAD z-score normalization,
  `FOREX_BOT_DISABLE_SMC_GATE`, threshold re-calibration on
  `random_coarse_threshold` to `[0.30..1.20]`.
- **Missing data:** per-gene `nonzero combined[i]` count. Without this
  instrumentation we are guessing which gate kills the signal.
- **Source:** v0.4.0 release notes "Still deferred to v0.5";
  `docs/audits/search_discovery_pipeline_audit_2026-05-03.md`.

### 2.2 Manual onboarding — the user has to do too much

- Env vars still required in practice (`FOREX_BOT_NORMALIZE_FEATURES`,
  `FOREX_BOT_DISCOVERY_MODE`, `FOREX_BOT_DISABLE_SMC_GATE`,
  `FOREX_BOT_DISCOVERY_PERMISSIVE`, `FOREX_BOT_SEARCH_EVAL_CUDA_DEVICE`,
  `FOREX_BOT_REQUIRE_GPU`, plus 7 prop-firm tuning knobs).
- `data/symbol_metadata.json` must exist or pip math is wrong.
- `config.yaml` must be hand-edited for non-default symbols.
- No first-run wizard. No "import my CSV folder, figure the rest out"
  one-button flow in the GUI (the CLI has `import`; the GUI does not).
- **Source:** `docs/audits/hardware_autodetect_config_ui_architecture_2026-05-03.md`,
  `docs/audits/rust_env_flags_config_debt_audit_2026-05-03.md`.

### 2.3 Config / UI / TUI drift

- `ResolvedConfig` (v0.4.0 P1) is the single source of truth on the
  backend, BUT three surfaces still read settings independently:
  - `forex-cli config` — uses `ResolvedConfig::display_table()` ✓
  - TUI **Config** page (key 8) — uses `ResolvedConfig::display_table()` ✓
  - `forex-app` Settings tab — reads its own `Settings` snapshot,
    does NOT go through `ResolvedConfig`. ✗
- Hidden env knobs (`FOREX_BOT_NORMALIZE_FEATURES`,
  `FOREX_BOT_DISABLE_SMC_GATE`, `FOREX_BOT_DISCOVERY_MODE`) are visible
  in CLI/TUI but not yet exposed as editable controls in `forex-app`.
- **Source:** `docs/audits/core_config_domain_modularization_audit_2026-05-04.md`.

### 2.4 Dead code + duplication

- ~50 clippy warnings remain (mostly `dead_code`, "casting to the same
  type", unused theme constants).
- 5 "god files" still 90–153 KB:
  `forex-models/training_orchestrator.rs` (153 KB),
  `forex-app/app_services/trading.rs` (132 KB),
  `forex-models/forecasting/swarm_impl.rs` (117 KB),
  `forex-search/discovery.rs` (110 KB),
  `forex-models/rl/dqn_impl.rs` (104 KB).
- 14 retained `#[allow(dead_code)]` blocks documented in
  `docs/audits/dead_code_allowlist_2026-05-10.md` — debt is explicit
  but not yet paid.
- **Source:** `docs/audits/post_release_tech_debt_2026-05-12.md`,
  `docs/audits/dead_code_and_stale_artifacts_audit_2026-05-03.md`,
  `docs/audits/dedup_findings_2026-05-10.md`.

### 2.5 UI/UX — broken usability

From the static review at `docs/qa/2026-05-12-ui-ux-static-review.md`
and the live test at `docs/qa/2026-05-12-gui-test-report.md`:

- **UTC clock can freeze** when no engines are running (no
  `ctx.request_repaint_after()` in status bar render).
- **Headless job completion silent** — `_rx` dropped at
  `forex-app/main.rs:110`, so a 30-min discovery emits no top-level
  "DONE" line in the keep-alive log.
- **No top-bar Engine dropdown** (intentional design change; sidebar
  tabs replaced it) — but there is **no "Start everything" button**
  anywhere. The user must manually open Discovery tab → click Start,
  then Training tab → click Start.
- **Out-of-process headless jobs invisible** to a separately-launched
  GUI — no shared state file, no socket.
- **No dedicated Logs panel tab in GUI** — only the per-tab snapshot
  card. Operator must `tail -f logs/forex-ai.log` separately.
- **P7 workstation layout (TradingView-style)** never landed. Current
  layout is icon-rail sidebar + dock (v0.2.x state).

### 2.6 Distribution

- **No Linux build** in v0.4.0 (cross-compile from source only).
- No installer / MSI / DMG / `.deb`.
- No signed binaries.
- No sample data bundled — the user has to find CSVs from somewhere.
- README is dev-facing, not end-user facing.

### 2.7 Security debt

- **14 open dependabot vulnerabilities** (6 high, 1 moderate, 7 low) on
  origin per `docs/audits/post_release_tech_debt_2026-05-12.md`.
- Two open dependabot PRs were merged on 2026-05-11
  (`openssl-0.10.79`, `rustls-webpki-0.103.13`); the remaining 12 are
  unresolved.

## 3. Patch plan — `v0.4.1` through `v0.5.0`

Each patch is **independent and shippable**. The numbering reflects
order, not strict dependency — patches can be reordered or parallelised
if scope changes.

### `v0.4.1` — Discovery instrumentation + dead-code baseline ★

**Scope:**

1. **`signals_for_gene_full` instrumentation pass** in
   `crates/forex-search/src/discovery.rs`:
   - For every gene evaluated in the GA, count nonzero values of
     `combined[i]` (the raw combined-signal vector before thresholding).
   - Emit to a new `<symbol>_<tf>_gene_signals.jsonl` next to the
     existing funnel JSON. One line per gene with:
     `gene_signature_hash`, `nonzero_count`, `nan_count`,
     `min_abs`, `max_abs`, `passed_filter` boolean.
   - No behaviour change. Pure visibility.
2. **Dead-code baseline doc** at
   `docs/audits/v0.4.1_dead_code_baseline.md`:
   - Snapshot of all `#[allow(dead_code)]` sites with their crates +
     line numbers.
   - List of clippy warnings with `cargo clippy --workspace
     --all-targets --release --message-format=json`.
   - This is the contract that `v0.4.2`–`v0.4.5` must shrink.

**Acceptance:**

- Running `forex-cli auto-loop --symbols EURJPY,XAUUSD --timeframes H1`
  produces `cache/EURJPY_H1_gene_signals.jsonl` with one line per
  evaluated gene.
- `docs/audits/v0.4.1_dead_code_baseline.md` exists and lists ≥ all
  current `#[allow(dead_code)]` sites.
- `cargo clippy --workspace --all-targets --release` count of warnings
  does NOT regress vs v0.4.0.
- All 764 existing tests pass.

**Branch:** `claude/v0.4.1-discovery-instrumentation`
**Tag at end:** `v0.4.1`

### `v0.4.2` — Discovery yield root-cause fix

**Scope:** ingest the `gene_signals.jsonl` data from v0.4.1, identify the
bottleneck, and ship the right fix. Most likely candidates:

- NaN guard pass in `combined[i]` computation before threshold compare.
- Per-feature distribution-derived thresholds in
  `random_coarse_threshold` (instead of hardcoded levels).
- Tiered fallback: if the strict threshold yields 0 candidates, retry
  with a relaxed threshold and label the resulting genes
  `confidence_tier=relaxed`.

**Acceptance:**

- Default-config discovery on EURJPY M30 / H1 / H4 / D1 produces
  **≥ 10 candidate strategies** that pass `nonzero_signals` and
  `passed_min_trades`.
- Same for XAUUSD on the same timeframes.
- EURUSD / GBPUSD / AUDUSD yield ≥ what they did in v0.4.0 (no
  regression).
- Funnel JSON shows the bottleneck has moved past `nonzero_signals`.

**Branch:** `claude/v0.4.2-discovery-yield-fix`
**Tag at end:** `v0.4.2`

### `v0.4.6` — Zero-touch onboarding

**Scope:**

- **Auto-detect data path** in `forex-cli` and `forex-app`: probe
  `./data`, `%APPDATA%/forex-ai/data`, `~/.local/share/forex-ai/data`,
  in that order. If none exists, prompt to create.
- **First-run wizard** in both CLI and GUI:
  - CLI: interactive prompts on the first invocation; saves to
    `config.yaml`.
  - GUI: modal on first launch with the same questions.
  - Questions: data directory, broker (cTrader / None), default
    symbol(s) to discover, GPU yes/no.
- **Auto-import from sample bundle**: ship `samples/` directory in the
  release archive with one CSV per major pair, M1 history for 1 year.
  On first run, if `data/` is empty, prompt to import the bundle.
- **Auto-train trigger**: when a discovery run produces ≥ 1 strategy,
  emit a "Train this portfolio?" prompt (CLI) / banner (GUI). Click
  yes → launch training in-app, no separate command needed.
- **Single "Start everything" button** in `forex-app` top bar: runs
  `discover → train → validate → ready-to-trade` for the configured
  symbol set. Status visible in bottom bar.

**Acceptance:**

- Fresh OS install + `forex-cli` (with sample bundle) produces a
  non-empty `data/.../portfolio.json` on EURUSD within 5 minutes,
  zero manual config edits.
- GUI first-launch on the same fresh install reaches a populated
  chart + a working "Start everything" button within 60 s of opening.
- No env var is required for the golden path.

**Branch:** `claude/v0.4.6-zero-touch-onboarding`

## 4. Per-patch deliverable contract

Every `v0.4.x` patch must produce, in order:

1. **A branch** named `claude/v0.4.x-<topic>` from latest `master`.
2. **A draft PR** opened against `master` immediately on branch
   creation, before any commits, so progress is visible.
3. **At least one commit per scope item** in the patch, with a clear
   commit message.
4. **A checklist in the PR body** with the acceptance criteria from
   §3.
5. **A test run log** attached as a commit or PR comment:
   - `cargo test --workspace` count + pass/fail.
   - `cargo clippy --workspace --all-targets --release` warning count.
   - Manual checklist results where applicable.
6. **A tag** `v0.4.x` on `master` once the PR is merged.
7. **A release** on GitHub with the same tag, body summarising the
   patch's user-visible changes, and any binary artifacts updated.

If a patch cannot meet its acceptance criteria, it is **split into
sub-patches** (`v0.4.x.1`, `v0.4.x.2`) rather than shipped partial.

## 5. Risks and unknowns

### High-risk items

- **v0.4.2 discovery yield fix.** The root cause is genuinely not
  pinned down. The v0.4.1 instrumentation may reveal that the
  bottleneck is elsewhere (e.g. SMC gate, prop-firm gate, correlation
  rejection) rather than NaN propagation. The plan absorbs this — the
  fix is data-driven.

- **v0.4.5 god-file splits.** `training_orchestrator.rs` and
  `dqn_impl.rs` are dense state machines; splitting them without
  changing behaviour requires careful re-exports. Reserve the option
  to skip a split if the dedup gain is < 30%.

- **v0.4.6 first-run wizard in the GUI.** egui modal dialogs are not
  truly modal — they coexist with the rest of the workspace. May need
  to introduce a `FirstRunOverlay` that captures all input events.

### Medium-risk items

- **v0.4.9 signed binaries.** Requires a code-signing cert; if the
  user does not have one we ship unsigned + document the warning,
  which is a UX regression vs the goal.

- **v0.4.7 workstation layout.** egui_dock supports the structure but
  the resize behaviour on small screens (< 1366×768) is fragile. Need
  to test on a laptop before claiming "responsive".

### Low-risk items

- **v0.4.4 dead-code purge.** Mechanical. The retained
  `#[allow(dead_code)]` blocks are already documented; the risk is
  zero behaviour-change but reviewer fatigue on large diffs.

---

## 6. Out of scope for v0.5

These are explicitly deferred to v0.6 or later:

- **Multi-user / multi-tenant.** v0.5 ships as a single-user desktop
  binary. No server mode, no cloud deployment.
- **Live trading on real money accounts.** v0.5 supports cTrader live
  in code but the README will instruct users to stay on demo.
- **iOS / Android apps.** Use Claude Code Remote Control for phone
  access; no native mobile app planned.
- **Auto-update.** v0.5 users must download new releases manually.
- **Plugin system.** No third-party module API in v0.5.
- **Strategy marketplace.** No share/import flow between users.
